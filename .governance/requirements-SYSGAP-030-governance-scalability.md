## 第一步：问题澄清（5 问法）

### Q1: 当前痛点是什么？

software-project-governance 工作流的 4 个核心治理文件随项目时间线性膨胀，且每次会话强制读取全部内容。本项目运行仅 3 周、168 个任务，治理文件合计已达 **449 KB**（约 180K tokens，占 1M 上下文窗口的 18%）。如果项目持续 6-12 个月达到 1000 个任务，治理文件将膨胀至 ~2.6 MB（约 1,055K tokens），仅治理数据读取就已经**超过** 1M 上下文窗口上限——agent 在接收任何用户指令之前就已经因治理数据过载而行为退化。

### Q2: 为什么这是个问题？

**量化影响**：

| 规模 | 任务数 | plan-tracker | evidence-log | decision-log | risk-log | 合计 | 估计 tokens | 占 1M 窗口 |
|------|--------|-------------|-------------|-------------|---------|------|------------|-----------|
| 当前 (3 周) | 168 | 193 KB | 163 KB | 74 KB | 19 KB | 449 KB | ~180K | 18% |
| 6 个月 | 1,000 | ~1,150 KB | ~990 KB | ~430 KB | ~67 KB | ~2,637 KB | ~1,055K | **超标 5.5%** |
| 12 个月 | 2,000 | ~2,300 KB | ~1,970 KB | ~860 KB | ~135 KB | ~5,265 KB | ~2,106K | **超标 111%** |
| 24 个月 | 5,000 | ~5,750 KB | ~4,935 KB | ~2,150 KB | ~335 KB | ~13,170 KB | ~5,268K | **超标 427%** |

> **增长系数推导**：
> - plan-tracker: 193 KB / 168 tasks = ~1.15 KB/task（每 task 新增 2-3 行表格记录）
> - evidence-log: 164 KB / 237 条证据 = ~700 bytes/条证据（准确值 164,084 bytes / 237 条 = 692 bytes/条，取整 ~700）。**关键修正——条目增长率**：实际证据条目与任务的比例为 237 条 / 168 tasks = **1.41 entries/task**（部分 P0 大 task 按 Phase 分段产生多条证据记录）。v1 版隐含假设 1.0 entries/task 是错误的——这导致 evidence-log 在 1000-task 规模下低估约 36%（700 KB → 990 KB），总治理数据低估 12%，且误判"1000-task 接近窗口上限（94%）"而实际已超标 5.5%。**本修正版所有投影均采用 1.41 entries/task**
> - decision-log: 74 KB / 60 条决策 = ~1.23 KB/决策（约每 3 task 产生 1 条决策）
> - risk-log: 19 KB / 28 条风险 = ~679 bytes/风险（约每 6 task 产生 1 条风险）
>
> 增长基本是线性的（文件大小与任务/证据数成正比），无自动截断或归档机制。
> **Token 转换假设**：1 token ≈ 2.5 bytes。治理文件含中英混合内容——中文 token 密度为 1-2 chars/token（vs 英文 3-4 chars/token），实际 token 消耗可能高于此估算。本报告统一使用 2.5 bytes/token（偏乐观估计）。
> **修正结论**：1000-task 规模下治理数据 token 消耗已达 ~1,055K（超过 1M 理论窗口上限），而非原版声称的"接近上限（94%）"。问题比初版描述更紧迫——在 6 个月内就突破窗口上限。

**定性后果**：
- 每次会话启动，agent 读取治理数据的时间消耗占比越来越高，留给实际工作的时间/预算越来越少
- 治理文件的"信息密度"递减——大量已完成历史数据稀释了活跃数据
- Agent 在超大治理文件中检索特定信息（如"这个 task 的依赖是什么"）的效率随文件增大而下降

### Q3: 不解决会怎样？

1. **6 个月内治理体系自我坍塌**：当治理文件达到 ~2.6 MB 时，每次会话加载治理数据就需要 ~1,055K tokens——**已超过 1M 上下文窗口理论上限**。Claude Code 上下文窗口 1M tokens 为理论值，实际可用窗口更小（需留出空间给 SKILL 文件加载、用户输入、agent 输出）。届时 agent 要么截断读取导致治理信息不完整，要么因上下文耗尽而行为退化。
2. **治理从"助力"变为"负担"**：工作流设计的核心价值主张是"agent 在后台持续看护，用户专注思考而非流程管理"。但当治理文件本身成为 agent 的最大开销时，工作流违背了自己的设计目标。
3. **新项目 adopt 门槛急剧升高**：新用户初始化后，即使项目只有 10 个任务，也需要忍受一个庞大的治理框架（template 本身已有相当的固定开销）。
4. **verify_workflow.py 运行时开销线性增长**：当前 Python 脚本全文解析 4 个文件，随文件增大 I/O 和解析时间也会线性增长。

### Q4: 解决后的理想状态是什么？

每次会话启动时，agent 只读取"当前活跃上下文"（热数据），其大小与项目总任务数**脱钩**——无论项目是运行了 3 周还是 3 年，每次会话的治理读取开销保持在一个固定的、可预测的预算范围内（建议 < 50K tokens）。历史数据仍然可查询（通过索引或显式命令），但不强制每次加载。

### Q5: 怎么算成功？

| 指标 | 基线 (当前 168 tasks) | 目标值 | 验证方式 | 说明 |
|------|---------------------|--------|---------|------|
| 每次会话必读治理数据量 | ~449 KB (100% 全量) | **< 120 KB**（无论项目规模多大） | 测量 CLAUDE.md Step 1-2 bootstrap 实际读取的治理文件字节数 | **指标 1 = bootstrap 读取量**。这是 CLAUDE.md Step 1-2 指令触发后 agent 实际从磁盘读取的字节数——不是文件总大小。热数据文件中包含索引和当前活跃 task 的上下文（可能超过 bootstrap 实际需要的字段），所以热数据文件总大小可以 > 120 KB，但 bootstrap 实际读取量 < 120 KB |
| Token 预算占用比 | ~180K (18%) | **< 50K (< 5%)** | 基于必读数据量的 tokens 估算 | 依赖指标 1 的准确性——token 估算假设 1 token ≈ 2.5 bytes（含中英混合内容，偏乐观） |
| 1000-task 规模下的热数据文件总量 | 预估 ~2.6 MB（全量，修正后） | **< 500 KB（热数据文件总大小）+ 归档不限** | 构造 1000-task 场景测量 | **指标 3 = 热数据文件总大小**（plan-tracker 热部分 + evidence-log 热部分 + 索引文件 + 其他始终驻留的数据）。指标 3 > 指标 1，因为 bootstrap 不需要读热数据文件的所有字段（如 agent 可以跳过验证元数据、历史统计摘要等）；且索引文件的某些段可能仅在需要时查询。指标 1 是用户可感知的性能指标，指标 3 是架构设计的容量约束 |
| verify_workflow.py 全量检查 | 4 文件全文读取 | **功能不减**——所有现有 Check 仍然能通过（归档数据也在检查范围内） | 运行 `python verify_workflow.py check-governance` 全部 PASS | 明确可验证 |
| 人类可读性 | Markdown 文件，可直接阅读 | **保持**——归档/索引方案不破坏人类直接用编辑器打开阅读的能力 | 人工验证 | 归档文件仍然是独立可读的 Markdown，不需要工具即可浏览

| 9 | **迁移脚本** | 将本项目的当前治理文件按分层模型执行首次归档（验证方案可行性） |

### OUT（明确不做）

| # | 事项 | 理由 |
|---|------|------|
| 1 | **引入外部数据库（SQLite/PostgreSQL）** | Agent 必须在本地文件系统中读取治理数据——引入数据库会增加依赖、破坏人类可读性、且 Claude Code agent 不一定能直接查询数据库。约束条件明确要求"不能依赖外部数据库" |
| 2 | **放弃 Markdown 格式改用 YAML/JSON** | 人类可读性是硬约束。治理文件必须保持 Markdown 格式，让用户可以直接用编辑器打开阅读 |
| 3 | **自动删除旧数据** | 历史治理数据有查询价值（"这个决策当时为什么这么定"、"这个风险后来怎么样了"）。归档而非删除——归档文件仍然存在且可被查询 |
| 4 | **实时动态加载/懒加载机制** | Agent 读文件是静态操作，不支持"读到一半按需加载更多"。分层方案基于"读哪些文件"而非"读文件中的哪些段落"——因为 agent 的 Read 工具读取文件内容难以做段落级增量 |
| 5 | **改变治理文件的基础格式或表格结构** | 当前 verify_workflow.py 依赖特定的表格格式和 ID 命名规范（EVD-xxx, DEC-xxx, RISK-xxx）。改动格式 = 重写大量验证逻辑，风险过高。归档方案必须在保持现有格式的前提下进行 |
| 6 | **治理数据的压缩/二进制存储** | 违反人类可读性约束 |

### 关键假设（待验证）

| # | 假设 | 风险 | 验证计划 |
|---|------|------|---------|
| H1 | **已完成任务不需要每次会话读取**。当 task 标记为"已完成"且对应 evidence 已记录后，agent 不再需要感知该 task 的历史细节 | 中——某些情况下 agent 需要查询历史 task 的依赖关系或决策背景 | 在当前项目中随机选取 **20 个**已完成 task（~13% 样本量，覆盖 P0/P1/P2 不同优先级和不同阶段的 task），检查 CLAUDE.md Step 1-2 是否会因缺少这些 task 信息而做出错误判断。**错误判断的判定准则**：(a) agent 无法识别历史 task 间依赖链导致重复执行或遗漏；(b) agent 错误判定 Gate 状态（历史 task 的 evidence 缺失引发假阳性交叉验证告警）；(c) agent 在执行新 task 时触发了已被历史 task 解决过的风险。验证通过标准：20 个 task 中 0 个触发以上任何错误准则。此外，需考虑已完成 task 的数据特征一致性——已完成 task 均经过相同的 evidence 记录流程，已完成 population 的特征是均匀的，因此 20/154=13% 的样本量足以代表总体 |
| H2 | **归档可以按"版本"为边界**。一个版本发布后，该版本内所有 task + evidence + decision 可以整体归档 | 中——跨版本的依赖关系和决策引用可能断裂 | 检查当前 plan-tracker 中所有跨版本依赖（如 FIX-053 依赖 FIX-042 但可能跨版本），评估引用断裂风险。如果发现断裂，预案：(1) 跨版本依赖 task 留在热文件中延后归档；(2) 或归档后在索引中标注跨版本引用 |
| H3 | **decision-log 和 risk-log 不需要激进的归档**。这两个文件的增长速度远低于 plan-tracker 和 evidence-log（74 KB + 19 KB vs 356 KB），简单的时间分段（如年归档）可能就足够 | 低——但如果项目跑 3-5 年，它们也会成为问题 | 按当前增长率外推 3-5 年规模：decision-log 在 3 年后可能到 ~600 KB（假设 400 条决策），risk-log ~100 KB。确认届时是否仍可接受。**可接受标准**：单文件 < 200 KB 且 agent 读取时间 < 1 次 Read call（即不需要分片读取） |
| H4 | **索引文件可以保持小而高效**。一个轻量索引（task_id → 归档文件路径的映射）足以让 agent 和 verify_workflow.py 按需查询历史数据 | 低——索引本身的维护成本和一致性风险 | 构造方案原型后测量索引文件大小和维护复杂度。**量化标准**：1000-task 规模下索引文件 < 50 KB（约 50 bytes/entry × 1000 entries）且单次查询 < 100 bytes 额外读取 |
| H5 | **首次归档迁移可以自动化**。当前 168 task 的治理文件可以自动拆分为"活跃部分"+"归档部分"而不丢失信息 | 高——这涉及结构化的"读懂当前文件内容并正确拆分" | 必须先设计拆分规则，再手工验证首批归档结果，之后再考虑自动化。**验证通过标准**：(1) 归档后 verify_workflow.py check-governance 全部 PASS；(2) 归档前后 plan-tracker 的活跃 task 状态完全一致（条数、状态、依赖关系无变化）；(3) 归档文件中的 task 数量 + 热文件中的 task 数量 = 归档前 task 总数 |
| H6 | **Agent 能正确执行多文件读取策略**。当前 CLAUDE.md Step 1-2 从"读 1 个 plan-tracker.md + 1 个 evidence-log.md"变为"读热数据文件 + 按需索引查询归档文件"，agent 能否在跨会话里一致地执行这条新指令？（**WARNING 级风险补充——本假设未在 v1 报告中列明**） | **高**——CLAUDE.md Step 1-2 是项目最关键的执行路径。如果 agent 执行多文件读取时遗漏某个热数据文件或误解索引格式，可能导致：(a) agent 在不知情的情况下读取不完整的治理数据；(b) 错误判定 Gate 状态；(c) 跳过未被索引到的依赖 task。这是影响范围最广的假设——涉及每一次会话的启动流程 | **原型验证**：修改 CLAUDE.md Step 1-2 为多文件读取指令后，在 3 个不同场景中测试 agent 启动行为：(1) 项目刚归档后的首次会话（所有热数据干净）；(2) 归档后已有 5+ 新 task 的会话（热数据与归档数据混合）；(3) 需要查询归档数据以解决跨版本依赖的会话。每次测试检查 agent 是否正确报告 Gate 状态、依赖关系和风险数。验证通过标准：3 场景中 0 次遗漏关键治理信息。**注意**：此假设验证需在架构原型阶段执行，且验证失败意味着归档方案的 CLI 指令设计可能是错的——可能需要更简单的方案（如保留最小 plan-tracker 而非完全依赖索引） |

## 第四步：非功能需求初筛

| 维度 | 评估 | 说明 |
|------|------|------|
| **性能** | 重要 | 每次会话的治理文件读取量 < 120 KB（约 50K tokens），读取操作在 1-2 次 Read tool call 内完成 |
| **可靠性** | 关键 | 归档操作必须原子化——不能出现"task 从主文件移除了但归档文件中丢失"的情况。迁移脚本需要 dry-run + 验证模式 |
| **可维护性** | 重要 | 归档规则必须简单可解释——"版本发布后归档"或"已完成超过 N 天"等明确规则，而非启发式判断 |
| **兼容性** | 关键 | 必须向后兼容——已初始化的项目（已有 .governance/ 目录）可以通过迁移脚本完成归档，不影响现有治理数据 |
| **可扩展性** | 重要 | 当项目从 1000 task 增长到 10000 task 时，归档机制本身不应成为新的瓶颈（如索引文件线性膨胀） |
| **人类可读性** | 约束 | 归档文件仍然是 Markdown 格式，用户可以用编辑器打开阅读任何历史记录 |

## 第五步：FAQ（常见质疑与解答）

> **注**：本 FAQ 章节为审查修复（W5）补充——覆盖架构决策前必须回答的核心质疑。

### Q1: 归档后还能查到历史决策吗？查询路径是什么？

**答**：能。归档不等于删除——归档文件仍然是独立可读的 Markdown 文件，存储在 `.governance/archive/` 目录下。查询路径有两种：
1. **Agent 查询**：通过索引文件（`.governance/archive/index.md` 或 `index.json`）按 task_id/evidence_id/decision_id 定位归档文件，执行一次 Read tool call 获取内容。
2. **人类查询**：直接打开 `.governance/archive/` 目录，按版本号浏览归档文件——归档文件保持 Markdown 格式，无需工具即可阅读。

索引文件记录"哪个条目在哪个归档文件里"的映射，格式待架构阶段确定（候选：Markdown 表格 vs 轻量 JSON）。关键设计约束：索引查询不应超过 1 次额外的 Read tool call。

### Q2: 如何确保归档不丢失数据？

**答**：通过四层保障机制：
1. **原子化归档脚本**：归档操作使用"先写归档文件 -> 验证归档完整性 -> 再从主文件删除"的三步原子流程。中间任何一步失败时回滚。
2. **Dry-run 模式**：迁移脚本提供 `--dry-run` 模式——先模拟归档并输出差异报告，用户确认后再执行真实归档。
3. **Git 保护**：归档操作创建 commit——即使归档脚本出错，git checkout 即可恢复到归档前状态。
4. **verify_workflow.py 一致性检查**：归档后运行 `verify_workflow.py check-governance` 验证——task 总数、evidence 条目数、跨文件引用一致性必须与归档前完全匹配。

### Q3: 旧项目如何迁移？迁移失败怎么办？

**答**：迁移策略分两层：
1. **新项目**（初始化时无历史数据）：直接使用分层模板——`plan-tracker.md` 仅含当前活跃 task 的表格占位符，无需迁移。
2. **已有项目**（如本项目，已有 168 task / 449 KB 治理数据）：运行一次性迁移脚本——脚本自动分析现有 plan-tracker/evidence-log 内容，按版本边界拆分并生成归档文件。**迁移前自动创建 git commit 作为回滚点**。

迁移失败恢复路径（按优先级）：
1. `git checkout` 恢复迁移前的 commit（推荐——最安全）；
2. 从 `.governance/archive/` 备份目录手动复制回主文件（备选）；
3. 重新运行迁移脚本（前提：已修复导致失败的 bug）。

迁移脚本本身有幂等性要求——重复执行同一迁移不应产生重复归档条目。

### Q4: 归档会不会让 verify_workflow.py 变慢？

**答**：单次全量检查（`check-governance`）的 I/O 总量不变——原来读 4 个大文件，现在读 4+N 个小文件（热文件 + N 个归档文件）。但全量检查的执行频率远低于每次会话的 bootstrap 读取，所以性能瓶颈从"每次会话都慢"变为"偶尔的验证慢"——这正是我们希望达成的效果。

具体影响：
- **checks_evidence_completeness**：原来读 1 个 evidence-log.md -> 改为读热 evidence-log.md + N 个归档文件。执行时间可能略微增加（多文件 I/O），但总量不变。
- **checks_cross_references**：现在需要跨热文件和归档文件遍历条目。索引文件可以加速——先查索引定位目标归档文件，再定向读取，而非逐个打开所有归档文件。

目标：`verify_workflow.py check-governance` 全量检查的执行时间增量 < 50%（即从当前 ~2 秒到 < 3 秒）。这个性能指标应在架构阶段 (ADR) 中明确并验证。

### Q5: 如果归档过程中出错了怎么恢复？

**答**：归档操作为三阶段原子事务，每阶段有独立的错误处理和回滚策略：

| 阶段 | 操作 | 失败恢复 |
|------|------|---------|
| Phase 1: 写归档 | 从主文件提取数据写入 `.governance/archive/{version}/xxx.md` | 删除已写的不完整归档文件，主文件未修改——零损失 |
| Phase 2: 验证 | 对比归档文件内容与主文件中待移除内容的一致性（逐条比对） | 归档文件已写但验证不通过 -> 保留归档文件供人工检查，主文件暂不修改。agent 报告差异 |
| Phase 3: 提交 | 从主文件删除已归档的内容，更新索引 | git 回滚到 Phase 2 前的状态 |

如果 Phase 3 中 agent 崩溃（最坏情况）：主文件可能处于"部分删除"状态。恢复方案：(1) `git checkout` 恢复主文件；(2) 重新执行归档脚本（脚本检测到已有归档条目时自动跳过重复内容）。

**关键设计约束**（待架构阶段确定）：归档脚本 MUST 在 Phase 1 之前创建 git commit（回滚点）；归档脚本 MUST NOT 使用 `git reset --hard`（避免意外丢失未关联归档的修改）。

### Q6: 热/温/冷边界会不会随项目演变而失效？

**答**：会的——这是本方案必须正视的风险。边界的"漂移"有两种形式：
1. **热数据范围扩大**：项目复杂度增加导致"每次会话必读"的上下文变多（如活跃风险数增加、依赖链变长）。这是可管理的——热数据上限 < 120 KB 是硬指标，如果项目复杂度导致热数据膨胀逼近上限，说明需要重新评估"什么是真正的热数据"。
2. **版本节奏变化**：如果项目从密集开发期进入低频维护期（或反过来），版本发布频率变化会影响归档触发的节奏。解决方案：归档触发条件的优先级设计（见 IN #8）已内置兜底机制——即使没有版本发布，兜底的"N 天"触发条件也会防止数据无限累积。

**缓解措施**：
- 热数据上限 < 120 KB 作为持续监控指标——每次会话启动时 agent 应自我检查读取量是否超标。
- 温数据可以定期重新评估——每季度检查一次温数据中有多少应该转入冷数据归档。
- 如果项目规模远超当前预估（如 10000+ task），分层模型可能需要升级（如引入"深度归档"——但那是另一个 SYSGAP，不是本任务要解决的）。

**核心原则**：本方案追求的不是完美的热/温/冷边界定义——而是"比当前全量读取更好的模型"，且具有可演进性（边界参数可通过配置调整，而非硬编码在代码中）。

## 初步方案方向（供架构阶段参考）

基于上述分析，建议架构阶段 (ADR) 重点评估以下方案方向：

### 方向 A：版本归档（推荐作为主方案）

**核心思路**：以"版本发布"为归档边界。当一个版本发布（git tag）后，该版本内的所有 task、evidence、decision 从主治理文件归档到 `.governance/archive/{version}/` 目录下的独立 Markdown 文件。

**优势**：
- 边界清晰——版本发布是自然的分割点
- 与现有版本规划体系一致（V-Gate, 版本路线图）
- 归档文件仍然可读（按版本查历史很方便）

**挑战**：
- 跨版本依赖引用需要处理（task 可能依赖已归档版本中的 task）
- 版本之间的"进行中"task 如何处理
- 决策可能跨越多个版本——归档到哪个版本？

### 方向 B：时间窗口归档

**核心思路**：保留最近 N 天（如 30 天）的数据在主文件中，超过 N 天的自动归档。

**优势**：
- 简单直观
- 不依赖版本概念（适用于无严格版本管理的项目）

**挑战**：
- 时间边界不如版本边界清晰（"30 天"是任意选择）
- 可能与版本周期不匹配（一个版本可能跨 2 个月）

### 方向 C：热/温/冷三层 + 索引

**核心思路**：
- 热文件：`.governance/plan-tracker.md`（仅含活跃 task）+ `.governance/evidence-log.md`（仅含最近证据）
- 归档目录：`.governance/archive/` 下的历史文件
- 索引文件：`.governance/archive/index.json`——记录每个 task_id/evidence_id/decision_id 在哪个归档文件中

**优势**：
- 最灵活的查询能力（通过索引按需定位历史数据）
- verify_workflow.py 可以通过索引遍历所有数据

**挑战**：
- 索引文件本身的维护（需要与归档文件保持一致性）
- JSON 格式的索引破坏了"纯 Markdown"承诺（虽然索引主要是给机器读的）
- 索引也可能膨胀（但远慢于原始数据）

### 方向 D：混合方案（推荐深入评估）

**版本归档 + 索引**：
- 版本发布后归档该版本的所有数据
- 索引文件（JSON 或 Markdown 表格）记录映射关系
- 热文件只保留当前活跃版本（未发布）的数据
- 决策和风险评估是否也需要跟着归档
