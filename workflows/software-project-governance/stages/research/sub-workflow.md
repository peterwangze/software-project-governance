# 调研与竞争分析 — 子工作流

本文件定义"调研与竞争分析"阶段的独立子工作流。用户可单独加载本文件执行调研阶段。

## 阶段目标

通过调研和竞争分析建立决策事实基础。调研阶段的核心产出是"调研报告"——用事实和数据回答"外面有什么、别人怎么做的、我们该注意什么"。

## 独立使用时的目标锚定

> 当本子工作流被独立加载（非通过 `main-workflow.md` 总入口加载）时，**MUST** 在执行阶段活动前完成：

1. **目标确认**：读取 `.governance/plan-tracker.md`，确认项目立项时定义的量化成功标准
2. **偏离检查**：对照成功标准，确认调研方向与项目总目标一致。如发现偏离，**MUST** 先记录到 `.governance/decision-log.md` 再继续
3. **质量底线**：调研结论必须包含数据支撑，不接受纯定性结论（参考 profile 质量要求）

## 进入条件

- 立项阶段已完成（G1 通过）
- 已明确需要调研的问题域

## 活动清单

### 1. 调研问题定义

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 列出需要回答的问题 | 基于立项阶段的目标，明确调研要回答哪些问题 | 需用户输入：需要调研什么 |
| 确定调研范围 | 技术维度、市场维度、用户维度 | 需用户确认 |

### 2. 技术/市场/用户三维调研

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 技术调研 | 当前技术方案的成熟度、趋势和风险 | 自动：搜索公开资料、文档、社区讨论 |
| 市场/竞品调研 | 竞争格局、替代方案、差异化空间 | 自动：搜索竞品信息 |
| 用户/需求调研 | 目标用户的痛点、期望和使用场景 | 需用户输入：用户画像和场景 |

### 3. 竞争分析矩阵

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 列举竞品/替代方案 | 至少列举 3 个竞品或替代方案 | 自动搜索 + 用户补充 |
| 建立对比维度 | 功能、性能、成本、体验、生态等 | 需用户确认对比维度 |
| 填充对比矩阵 | 逐个竞品按维度评估 | 自动：基于搜索结果填充 |

### 4. 关键发现总结

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 提炼关键发现 | 从调研数据中提炼 3~7 条关键发现 | 自动 |
| 标注数据支撑 | 每条发现标注数据来源 | 自动 |
| 识别可行约束 | 列出技术可行性、市场窗口、资源约束 | 自动 + 需用户确认 |

### 5. 可行性评估

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 评估技术可行性 | 当前技术栈能否支撑目标 | 自动 |
| 评估市场可行性 | 市场窗口是否合适 | 需用户判断 |
| 评估资源可行性 | 团队能力和资源是否匹配 | 需用户判断 |

## 产出物标准

| 产出物 | 格式要求 | 质量标准 |
|--------|---------|---------|
| 调研报告 | 覆盖技术、市场、用户三个维度 | 每个维度至少 3 条有数据支撑的发现 |
| 竞争分析矩阵 | 至少 3 个竞品 × 至少 4 个对比维度 | 矩阵完整无空白 |
| 可行性评估 | 技术/市场/资源各有明确结论 | 结论有数据支撑，不是猜测 |
| 关键发现总结 | 3~7 条发现，每条有数据来源 | 发现之间不重复、不矛盾 |

## 退出条件

- [ ] 调研覆盖了技术、市场和用户三个维度
- [ ] 竞争格局已清晰（至少 3 个竞品的对比矩阵）
- [ ] 关键发现有数据支撑（每条发现标注来源）
- [ ] 可行性有初步判断（技术/市场/资源各有结论）

## Gate 映射

本阶段对应 **G2 — 调研完成**。

| Gate 检查项 | 对应活动 | 判定标准 |
|------------|---------|---------|
| 调研是否覆盖了技术、市场和用户三个维度？ | 三维调研 | 三个维度各有至少 3 条发现 |
| 竞争格局是否清晰？ | 竞争分析矩阵 | 矩阵至少 3×4，无空白项 |
| 关键发现是否有数据支撑？ | 关键发现总结 | 每条发现标注了数据来源 |
| 是否识别了技术可行性的关键约束？ | 可行性评估 | 至少 1 条技术约束，有明确影响描述 |
