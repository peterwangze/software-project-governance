# Lifecycle 阶段详细定义

本文件供 agent 在进入特定阶段时按需读取。SKILL.md 已内嵌阶段列表摘要和治理规则。

## 1. 立项与目标定义（initiation）

- **目标**：明确问题、目标、边界、成功标准和关键约束
- **核心活动**：问题陈述、目标定义、范围边界、成功标准、干系人识别、初始风险评估
- **产出物**：项目章程、目标说明、范围边界文档、初始风险清单
- **退出条件**：目标可衡量、范围有边界、干系人已识别、初始风险已记录
- **子工作流**：`stages/initiation/sub-workflow.md`（如在插件环境中可用）

## 2. 调研与竞争分析（research）

- **目标**：通过调研和竞争分析建立决策事实基础
- **核心活动**：市场/技术调研、竞争产品分析、用户需求分析、可行性评估
- **产出物**：调研报告、竞争分析矩阵、可行性评估、关键发现总结
- **退出条件**：调研覆盖关键决策维度、竞争格局清晰、可行性结论明确

## 3. 技术选型与方案预研（selection）

- **目标**：基于调研结果做出技术选型和方案方向决策
- **核心活动**：候选方案列举、评估标准定义、PoC/原型验证、方案对比、决策留痕
- **产出物**：选型评估表、PoC 结果、技术方案决策记录、风险评估更新
- **退出条件**：技术方向已选定、关键风险已验证、决策已记录

## 4. 环境搭建与基础设施（infrastructure）

- **目标**：建立项目开发、测试、部署的基础设施
- **核心活动**：开发环境配置、代码仓库初始化、CI/CD 基础搭建、工具链选型、权限与规范
- **产出物**：环境配置文档、仓库结构、CI/CD 基础配置、工具链说明
- **退出条件**：开发环境可用、仓库结构已建立、基础 CI 可运行

## 5. 架构设计（architecture）

- **目标**：形成可执行的技术方案和系统设计
- **核心活动**：系统架构设计、模块划分、接口定义、概要设计、详细设计、技术评审
- **产出物**：架构设计文档、模块划分图、接口定义、概设/详设文档、技术评审结论
- **退出条件**：架构评审通过、关键接口已定义、详细设计覆盖核心模块

## 6. 开发实现（development）

- **目标**：按设计产出可工作的软件资产
- **核心活动**：迭代开发、代码编写、单元测试、Code Review、集成调试
- **产出物**：源代码、单元测试、Code Review 记录、集成验证结果
- **退出条件**：核心功能已实现、单元测试覆盖达标、Code Review 遗留项已关闭

## 7. 测试与质量保障（testing）

- **目标**：验证软件质量达到发布标准
- **核心活动**：集成测试、系统测试、性能测试、安全测试、回归测试、缺陷管理
- **产出物**：测试计划、测试用例、测试报告、缺陷清单、质量评估
- **退出条件**：关键缺陷已关闭、回归测试通过、性能指标达标

## 8. 防护网与 CI/CD（ci-cd）

- **目标**：建立持续集成、持续交付的质量防护网
- **核心活动**：CI 流水线完善、自动化测试集成、代码质量门禁、部署流水线、监控告警
- **产出物**：CI/CD 配置、自动化测试套件、质量门禁规则、部署文档
- **退出条件**：CI 流水线稳定、自动化测试覆盖核心路径、质量门禁生效

## 9. 版本发布（release）

- **目标**：按计划发布版本并确保发布过程可控
- **核心活动**：发布计划、变更清单、发布测试、灰度/金丝雀发布、回滚方案
- **产出物**：发布计划、变更日志、发布 checklist、回滚方案、发布验证结果
- **退出条件**：发布验证通过、监控指标正常、回滚方案已验证

## 10. 运营与反馈（operations）

- **目标**：跟踪实际运行状况、收集反馈、识别改进点
- **核心活动**：运营监控、用户反馈收集、问题归档、性能跟踪、优化建议
- **产出物**：运营报告、反馈汇总、问题清单、优化建议
- **退出条件**：首轮运营数据已收集、关键问题已归档、优化方向已明确

## 11. 维护与演进（maintenance）

- **目标**：基于复盘和反馈持续修订项目资产
- **核心活动**：问题修复、规则修订、模板更新、版本规划、复盘会议
- **产出物**：修复记录、规则修订、模板更新、复盘结论、下轮规划
- **退出条件**：复盘完成、修订已回灌、下轮方向已明确

## 阶段关系

### 可重叠阶段

**受 profile 约束**：strict profile 禁止阶段重叠（见 `profiles.md`），standard/lightweight 允许以下相邻阶段重叠：

- 调研（2）与技术选型（3）：调研中期即可开始候选方案评估
- 环境搭建（4）与架构设计（5）：基础环境先行，详细设计可迭代
- 开发（6）与测试（7）：测试用例编写可与开发并行
- CI/CD（8）与发布（9）：CI 就绪后可开始发布准备
- 防护网（8）与版本发布（9）：CI 稳定后即可准备发布流程

### 回退规则

- 回退必须更新风险记录和决策记录
- 回退不意味着"推翻已完成"，而是"补齐缺失"
- 回退范围由项目 profile 决定

## 触发模式

- **always-on**：每次会话自动加载，持续跟踪状态。适用：正式项目。
- **on-demand**：用户主动触发。适用：已有成熟流程的项目。
- **silent-track**：后台记录变更，只在 Gate 时提醒。适用：个人项目。

## 项目治理规则

1. 目标看护：初始目标在立项阶段定义，后续偏离必须更新决策和风险记录
2. 阶段推进：推进前必须通过 Gate 检查，状态变更必须回写事实源
3. 过程可信：已完成事项必须绑定证据，关键取舍必须进入决策记录
4. 统一要求：所有治理记录集中写入 `.governance/`，不得维护不同版本
5. 审计看护：所有阶段必须接受审计（参考 `references/audit-framework.md`）。审计不是可选动作——Gate 通过前必须执行适用的审计维度。审计发现必须转化为 plan-tracker/decision-log/risk-log 条目。审计机制本身也接受审计（Meta-Audit：发现→任务转化率 ≥ 80%，连续 3 次审计无发现 → 审计维度可能需要调整）。
