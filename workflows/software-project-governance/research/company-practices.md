# 公司实践到工作流规则映射

本文件将大型软件公司的项目管理经验拆解为三层：经验、规则、插件行为。

## 1. 单一真实来源

### 经验

大型软件团队会将任务状态、责任人、优先级和里程碑维护在统一入口，避免多套事实源失真。

### 规则

- 每个 workflow 必须定义唯一主计划入口。
- 不同 agent 不得维护不同版本的主计划。

### 插件行为

- 进入任务执行前，agent 必须先读取计划主入口。
- 更新状态时必须回写同一份计划记录。

## 2. 阶段门禁（Stage Gate）

### 经验

大型软件公司通常使用阶段评审点控制项目推进质量。

### 规则

- 每个阶段必须定义进入条件、退出条件和 Gate。
- 未通过 Gate 不得进入下一阶段。

### 插件行为

- agent 在阶段切换前必须检查 Gate。
- 如果 Gate 未通过，应阻断推进并要求补齐缺口。

## 3. 证据驱动

### 经验

成熟组织强调交付结果必须有证据支撑，而不是口头宣称完成。

### 规则

- 已完成事项必须至少绑定一条证据。
- Gate 结论必须可追溯到证据。

### 插件行为

- agent 声称任务完成前，必须确认已补齐证据记录。
- 缺少证据时，应将事项保留在未闭环状态。

## 4. 决策留痕

### 经验

大型组织要求关键决策具备回溯性，便于维护和复盘。

### 规则

- 关键取舍、范围变化、门禁变化必须进入决策记录。

### 插件行为

- 遇到重大方案切换时，agent 应更新决策记录而不是只改结果文档。

## 5. 风险前置

### 经验

成熟团队在规划早期显式记录风险和阻塞项，并持续跟踪缓解动作。

### 规则

- 阻塞项必须记录为风险。
- 风险状态变化必须回写主计划。

### 插件行为

- agent 识别到阻塞时，应同步更新风险记录。
- 若风险未闭环，不应将任务标记为完全完成。

## 6. 持续改进

### 经验

大型组织会把项目复盘经验回灌到模板、规范和流程本身。

### 规则

- workflow 必须保留样例与复盘入口。
- 模板和规则必须可演进。

### 插件行为

- agent 完成阶段性工作后，应同步更新样例和经验沉淀。
