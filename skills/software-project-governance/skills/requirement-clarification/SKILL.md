# 需求澄清 Checklist

本 skill 用于立项阶段的需求澄清，确保项目目标、范围和约束在启动时已充分对齐。可独立使用，不依赖完整生命周期。

## 触发条件

- 项目刚刚启动，需要明确要做什么
- 用户提出模糊需求（如"做一个XX系统"、"优化性能"）
- 立项子工作流执行到"问题陈述"或"目标定义"环节

## 输入

| 输入项 | 说明 | 是否必须 |
|--------|------|---------|
| 初始需求描述 | 用户对项目的初步想法 | 必须 |
| 业务背景 | 项目所在的业务领域和上下文 | 推荐 |
| 约束条件 | 时间、预算、技术栈等已知限制 | 推荐 |

## 执行步骤

### 第一步：问题澄清（5 问法）

依次回答以下问题，直到每个问题都有明确答案：

| # | 问题 | 目的 | 通过标准 |
|---|------|------|---------|
| 1 | 当前痛点是什么？ | 确认问题真实存在 | 用 1~3 句话描述现状问题，不包含解决方案 |
| 2 | 为什么这是个问题？ | 理解问题的影响 | 量化影响（时间损失、成本、质量下降）或定性说明后果 |
| 3 | 不解决会怎样？ | 评估紧迫性 | 说明不解决的代价，与解决问题形成对比 |
| 4 | 解决后的理想状态是什么？ | 定义目标 | 描述成功后的场景，不含具体技术方案 |
| 5 | 怎么算成功？ | 量化成功标准 | 至少 1 个可量化的指标（数字、百分比、时间约束） |

**判定规则**：
- 5 个问题全部有明确答案 → 进入下一步
- 问题 1 或 5 无法回答 → 需要用户补充信息，暂停并收集
- 问题 2~4 无法回答 → 标记为"待调研"，不阻塞立项，但在调研阶段必须补齐

### 第二步：范围澄清（IN/OUT 列表）

| # | 活动 | 通过标准 |
|---|------|---------|
| 1 | 列出范围内事项（IN） | 每条是一个具体的功能或交付物，不是泛化描述 |
| 2 | 列出范围外事项（OUT） | 至少 1 条"看起来应该做但决定不做"的事项 |
| 3 | 检查 IN/OUT 无歧义 | 任何一个事项不能同时出现在 IN 和 OUT 中 |
| 4 | 检查 OUT 有理由 | 每条 OUT 事项有"为什么不做"的说明 |

**参考 Amazon Working Backwards**：范围外列表必须包含至少 1 条"看起来应该做但决定不做"的事项，这迫使团队做出真实的优先级取舍。

### 第三步：干系人与约束澄清

| # | 问题 | 目的 | 通过标准 |
|---|------|------|---------|
| 1 | 谁关心这个项目的结果？ | 识别干系人 | 至少列出 1 个干系人，每条有角色和职责 |
| 2 | 有什么硬约束？ | 时间、预算、技术栈、合规 | 列出所有已知约束，无约束则标注"暂无" |
| 3 | 有什么假设？ | 识别未验证的前提 | 列出所有假设，无假设则标注"暂无" |
| 4 | 与其他项目/系统的关系？ | 识别依赖 | 列出依赖项，无依赖则标注"独立项目" |

### 第四步：非功能需求初筛

| 维度 | 问题 | 何时深入 |
|------|------|---------|
| 性能 | 有响应时间或吞吐量要求吗？ | 有具体指标时进入架构阶段详细设计 |
| 安全 | 涉及敏感数据或权限控制吗？ | 涉及时在技术选型阶段评估 |
| 可用性 | 有 SLA 要求吗？ | 有明确 SLA 时纳入非功能需求 |
| 可维护性 | 预期生命周期多长？ | 长期项目需在架构阶段考虑可维护性 |
| 兼容性 | 需要兼容现有系统吗？ | 有兼容需求时在技术选型阶段评估 |

**说明**：本步骤只做初步识别，不需要立即给所有维度定量化指标。识别出的非功能需求在后续阶段逐步细化。

## 输出

| 产出物 | 格式 |
|--------|------|
| 问题陈述 | 1~3 句话描述痛点 + 影响量化 |
| 成功标准 | 至少 1 个可量化指标 |
| IN/OUT 范围表 | 列表，OUT 至少 1 条有理由 |
| 干系人列表 | 每条有角色和职责 |
| 约束和假设列表 | 已知约束 + 未验证假设 |

## 质量标准

- [ ] 问题陈述不包含解决方案描述
- [ ] 成功标准至少 1 个可量化
- [ ] IN/OUT 列表无重叠，OUT 至少 1 条有理由
- [ ] 干系人列表非空，每条有角色描述
- [ ] 约束和假设已列出（可为"暂无"）

## 独立使用说明

本 skill 可脱离完整生命周期独立使用。

### 如果项目已初始化 governance（`.governance/` 存在）
1. 先执行目标锚定检查（读 plan-tracker → 确认成功标准 → 偏离检查）
2. 不需要先通过任何 Gate
3. 输出直接作为项目启动文档的素材
4. 如果后续接入完整工作流，本 skill 的产出自动成为立项阶段的证据

### 如果项目未初始化 governance（`.governance/` 不存在）
1. 跳过所有治理前置检查
2. 直接执行本 skill 的核心流程（5 问法 + IN/OUT 范围表）
3. 执行完毕后提醒用户："当前项目未初始化 governance。如需自动证据记录、Gate 检查、风险跟踪，运行 `/governance-init`。"

## 子工作流映射

| 本 skill 步骤 | 对应子工作流活动 | 映射说明 |
|--------------|-----------------|---------|
| 问题澄清 | 立项→问题陈述 | 5 问法是"描述当前痛点"+"量化问题影响"的结构化执行方式 |
| 范围澄清 | 立项→范围边界 | IN/OUT 列表直接映射为范围边界的产出物 |
| 干系人与约束 | 立项→干系人识别 | 输出的干系人列表即为 Gate G1 要求的干系人证据 |
| 非功能需求初筛 | 立项→初始风险评估 | 初筛结果中涉及安全/性能的部分可能成为初始风险项 |
