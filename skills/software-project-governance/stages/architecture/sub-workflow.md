# 架构设计 — 子工作流

本文件定义"架构设计"阶段的独立子工作流。用户可单独加载本文件执行架构设计阶段。

## 阶段目标

形成可执行的技术方案和系统设计。架构设计阶段的核心产出是"架构文档"——回答"系统怎么拆、接口怎么定、数据怎么流、非功能需求怎么满足"。

## 进入条件

- 技术选型已完成（G3 通过）
- 开发环境已搭建（G4 通过，可重叠）
- 有明确的调研结论和选型决策支撑

## 活动清单

### 1. 需求到设计的映射

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 梳理功能需求 | 将立项和调研阶段的产出映射为功能需求列表 | 需用户确认：需求优先级 |
| 梳理非功能需求 | 性能、安全、可用性、可维护性等 | 需用户输入：非功能需求的优先级 |
| 确认设计范围 | 哪些需求纳入首版设计 | 需用户确认 |

### 2. 系统架构设计

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 确定架构风格 | 单体/微服务/事件驱动/分层等 | 自动：基于需求特征推荐，需用户确认 |
| 模块划分 | 按职责拆分模块，定义模块边界 | 自动：输出候选方案，需用户确认 |
| 接口定义 | 模块间接口的输入输出和契约 | 自动：基于模块划分输出 |
| 数据流设计 | 数据在模块间如何流转和存储 | 自动 |

### 3. 概要设计

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 模块职责描述 | 每个模块做什么、不做什么 | 自动 |
| 依赖关系图 | 模块间的依赖和调用关系 | 自动 |
| 关键流程图 | 核心业务流程的端到端路径 | 自动 |

### 4. 详细设计

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 核心模块详细设计 | 核心模块的内部结构、算法、数据结构 | 自动 |
| 接口详细规范 | 接口的字段、类型、校验规则、错误码 | 自动 |
| 数据模型 | 表结构或数据 schema | 自动 |

### 5. 技术评审

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 自检清单 | 架构是否满足非功能需求、模块是否职责单一 | 自动 |
| 评审结论 | 通过/有条件通过/需要修改 | 需用户确认 |

## 产出物标准

| 产出物 | 格式要求 | 质量标准 |
|--------|---------|---------|
| 架构文档 | 包含架构风格、模块划分、接口定义 | 非功能需求有对应设计支撑 |
| 模块划分图 | 每个模块职责单一、边界清晰 | 无循环依赖 |
| 接口定义 | 输入输出、类型、校验规则完整 | 接口可独立测试 |
| 概设/详设 | 核心模块有详细设计 | 覆盖主路径 |

## 退出条件

- [ ] 架构评审通过或条件通过
- [ ] 模块划分清晰、职责单一（无循环依赖）
- [ ] 关键接口已定义（输入输出和类型完整）
- [ ] 详细设计覆盖核心模块

## Gate 映射

本阶段对应 **G5 — 设计完成**。

| Gate 检查项 | 对应活动 | 判定标准 |
|------------|---------|---------|
| 架构是否满足非功能性需求？ | 系统架构设计 | 每个非功能需求有对应的设计措施 |
| 模块划分是否清晰、职责是否单一？ | 系统架构设计 | 模块间无循环依赖，每个模块职责描述 ≤ 3 句话 |
| 关键接口是否已定义？ | 接口定义 | 核心接口有完整的输入输出和类型定义 |
| 是否经过技术评审？ | 技术评审 | 有评审结论（通过/条件通过/需修改） |
| 详细设计是否覆盖核心模块？ | 详细设计 | 核心模块有内部结构描述 |
