# 开发实现 — 子工作流

本文件定义"开发实现"阶段的独立子工作流。用户可单独加载本文件执行开发阶段。

## 阶段目标

按设计产出可工作的软件资产。开发阶段的核心产出是"可工作的代码"——通过单元测试、Code Review、集成验证的源代码。

## 进入条件

- 架构设计已完成（G5 通过或条件通过）
- 开发环境已搭建（G4 通过）
- 有明确的详细设计作为输入

## 活动清单

### 1. 开发准备

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 确认开发范围 | 本轮迭代实现哪些功能 | 需用户确认：优先级排序 |
| 创建开发分支 | 从主分支拉取开发分支 | 自动 |
| 确认编码规范 | 语言、命名、注释、提交信息格式 | 自动：检查项目已有规范，缺失则推荐 |

### 2. 迭代开发

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 按模块实现 | 按详细设计逐模块编码 | 自动 |
| 编写单元测试 | 每个功能模块配套单元测试 | 自动 |
| 自测通过 | 本地运行单元测试确保通过 | 自动 |

### 3. Code Review

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 自检清单 | 代码是否符合规范、是否有安全隐患、是否有性能问题 | 自动 |
| 提交 Review | 创建 Review 请求 | 自动 |
| 处理 Review 意见 | 修改代码并回复意见 | 自动：处理明确意见；需用户判断：架构级建议 |

### 4. 集成验证

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 合并到集成分支 | 将开发分支合并到集成环境 | 自动 |
| 运行集成测试 | 确保模块间交互正常 | 自动 |
| 修复集成问题 | 处理合并冲突和集成失败 | 自动 |

## 产出物标准

| 产出物 | 格式要求 | 质量标准 |
|--------|---------|---------|
| 源代码 | 符合编码规范、有必要的注释 | 静态检查通过 |
| 单元测试 | 覆盖核心逻辑路径 | 覆盖率达标（standard: ≥70%, strict: ≥90%） |
| Code Review 记录 | 所有 Review 意见已处理或说明 | 遗留项为 0 |
| 集成验证结果 | 集成测试全部通过 | 无阻塞性失败 |

## 退出条件

- [ ] 核心功能按设计实现（功能列表与设计文档对照无遗漏）
- [ ] 单元测试覆盖达标（standard ≥70%, strict ≥90%）
- [ ] Code Review 遗留项已关闭（遗留项 = 0）
- [ ] 集成验证通过（无阻塞性失败）

## Gate 映射

本阶段对应 **G6 — 开发完成**。

| Gate 检查项 | 对应活动 | 判定标准 |
|------------|---------|---------|
| 核心功能是否按设计实现？ | 迭代开发 | 功能列表与设计文档逐项对照，无遗漏 |
| 单元测试覆盖率是否达标？ | 单元测试 | coverage 报告 ≥ profile 要求的阈值 |
| Code Review 遗留项是否已关闭？ | Code Review | Review 工具中遗留项计数 = 0 |
| 集成验证是否通过？ | 集成验证 | 集成测试套件全部通过，无阻塞性失败 |
