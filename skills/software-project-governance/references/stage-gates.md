# Stage Gate 详细检查项

本文件供 agent 在执行 Gate 检查时按需读取。SKILL.md 已内嵌 Gate 通过类型和裁剪规则。

## G1 — 立项完成（立项 → 调研）

- **必审材料**：项目章程、目标说明、范围边界、初始风险清单
- **检查项**：
  1. 项目目标是否可衡量？→ 成功标准包含至少 1 个量化指标
  2. 范围边界是否清晰？→ 范围外事项列表非空
  3. 关键干系人是否已识别？→ 干系人列表 ≥ 1 条，每条有角色描述
  4. 是否有明确的"不做什么"清单？→ 范围外事项与范围内不重叠

## G2 — 调研完成（调研 → 技术选型）

- **必审材料**：调研报告、竞争分析矩阵、关键发现总结
- **检查项**：
  1. 调研覆盖技术、市场和用户三维度？→ 三个独立章节
  2. 竞争格局清晰？→ 竞品对比矩阵 ≥ 3 竞品 × ≥ 4 维度
  3. 关键发现有数据支撑？→ 每条发现标注数据来源
  4. 技术可行性约束已识别？→ 至少 1 条技术约束含影响描述和严重级别

## G3 — 选型完成（技术选型 → 环境搭建）

- **必审材料**：选型评估表、PoC 结果、技术方案决策记录
- **检查项**：
  1. 评估了至少 2 个候选方案？→ 选型评估表列出 ≥ 2 个
  2. 评估标准事先定义？→ 标准在评估前已定义（时间戳可验证）
  3. 选择原因已留痕？→ 决策记录有选择原因、备选方案和排除理由
  4. 关键风险已通过 PoC 验证？→ PoC 结果文档存在且结论为"验证通过"或"风险可控"

## G4 — 环境就绪（环境搭建 → 架构设计）

- **必审材料**：环境配置文档、仓库结构、基础 CI 配置
- **检查项**：
  1. 开发环境可复现？→ 有 Dockerfile/环境配置文档/一键脚本
  2. 仓库结构符合约定？→ 验证脚本通过
  3. 基础 CI 可运行？→ CI 配置文件存在且运行成功
  4. 协作规范已建立？→ 贡献指南/编码规范/提交规范至少 1 份

## G5 — 设计完成（架构设计 → 开发实现）

- **必审材料**：架构文档、模块划分、接口定义、概设/详设、技术评审结论
- **检查项**：
  1. 架构满足非功能性需求？→ 有非功能需求对应表
  2. 模块划分清晰、职责单一？→ 每个模块职责 ≤ 3 句话，无循环依赖
  3. 关键接口已定义？→ 核心接口有完整输入输出字段和类型
  4. 经过技术评审？→ 有评审结论（通过/条件通过/需修改）
  5. 详细设计覆盖核心模块？→ 核心模块有内部结构描述

## G6 — 开发完成（开发实现 → 测试）

- **必审材料**：源代码、单元测试、Code Review 记录、集成验证结果
- **检查项**：
  1. 核心功能按设计实现？→ 功能列表与设计文档逐项对照无遗漏
  2. 单元测试覆盖达标？→ coverage ≥ profile 阈值（lightweight: 无要求, standard: ≥70%, strict: ≥90%）
  3. Code Review 遗留项关闭？→ 遗留项计数 = 0
  4. 集成验证通过？→ 集成测试套件全部通过

## G7 — 测试通过（测试 → 防护网与 CI/CD）

- **必审材料**：测试计划、测试报告、缺陷清单、质量评估
- **检查项**：
  1. 关键缺陷已关闭？→ 关键/阻塞级别缺陷数 = 0
  2. 回归测试通过？→ 通过率 = 100%
  3. 性能指标达标？→ 响应时间/吞吐量/资源占用在目标范围内
  4. 安全测试覆盖关键风险？→ 覆盖 OWASP Top 10 关键项，无高危漏洞

## G8 — 防护网就绪（防护网与 CI/CD → 版本发布）

- **必审材料**：CI/CD 配置、自动化测试套件、质量门禁规则
- **检查项**：
  1. CI 流水线稳定？→ 最近 5 次运行成功率 ≥ 80%
  2. 自动化测试覆盖核心路径？→ 路径列表与设计文档功能模块对应
  3. 质量门禁生效？→ 规则文件存在且 CI 日志中可见门禁检查
  4. 部署流程文档化？→ 部署文档存在且步骤可执行

## G9 — 发布就绪（版本发布 → 运营）

- **必审材料**：发布计划、变更日志、发布 checklist、回滚方案
- **检查项**：
  1. 发布范围明确？→ 含版本号、发布范围、时间窗口
  2. 变更日志完整？→ commit 列表与 changelog 对照一致
  3. 回滚方案已验证？→ 有具体步骤且已在测试环境验证
  4. 发布后验证已定义？→ checklist 存在且包含核心功能验证项

## G10 — 运营回收（运营 → 维护）

- **必审材料**：运营报告、反馈汇总、问题清单、优化建议
- **检查项**：
  1. 收集到真实运营数据？→ 报告含至少 1 周真实运行数据
  2. 用户反馈已归档？→ 反馈条数 ≥ 0，有分类
  3. 关键问题已识别分类？→ 问题清单存在且每条有严重级别和状态
  4. 优化方向已明确？→ 至少 1 条可执行改进方向

## G11 — 维护闭环（维护 → 下一轮）

- **必审材料**：复盘结论、规则修订、模板更新
- **检查项**：
  1. 复盘完成？→ 文档含目标回顾、结果评估、原因分析、经验沉淀
  2. 经验回灌到规则和模板？→ 有对应更新提交记录
  3. 下轮方向已明确？→ 计划跟踪表中有 ≥ 1 条 P0 任务
  4. 版本化记录已更新？→ 版本/日期字段已更新

## Gate 执行原则

1. 未通过 Gate，不得声称进入下一阶段（受 profile 调节）
2. Gate 结论必须能回溯到证据和事实源
3. 任何 agent 适配都不得绕过 Gate 规则
4. Gate 检查可以由 agent 自动执行（可程序化部分）或由用户确认（需人工判断部分）
5. Gate 失败后的重新尝试不需要"回到上一阶段"，只需补齐缺失项后重新检查
6. **任务关闭的后续承诺追踪（Closure Follow-Through）**：当任务状态标记为"已完成"时，必须检查该任务的"备注"和"纠偏动作"列中是否包含对后续工作的承诺（如"再规划 XX 实现路线""后续需创建 YY 任务"）。如果存在此类承诺但 plan-tracker 中没有对应的实施任务，该任务不得标记为关闭——必须先创建后续任务或将承诺转化为明确的跟踪条目。此规则防止"备注里写了但从未执行"的治理漏洞。
