# 技术评审 Checklist

本 skill 用于架构设计阶段的技术评审，确保设计方案在提交 Gate 前经过系统化的质量检查。可独立使用。

## 触发条件

- 架构设计文档已完成，需要提交评审
- 概要设计或详细设计完成一个重要模块
- 用户要求对当前设计进行审核

## 输入

| 输入项 | 说明 | 是否必须 |
|--------|------|---------|
| 架构/设计文档 | 待评审的设计产出物 | 必须 |
| 非功能需求列表 | 性能、安全、可用性等要求 | 必须 |
| 技术选型决策记录 | 为什么选这些技术 | 推荐 |
| 已知约束 | 时间、资源、技术限制 | 推荐 |

## 执行步骤

### 第一步：结构完整性检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 有明确的设计目标吗？ | 目标与立项阶段的项目目标一致，且增加了技术层面的具体要求 |
| 2 | 有方案描述吗？ | 包含架构风格、模块划分、数据流、接口定义 |
| 3 | 有替代方案吗？ | 至少列出 1 个被否决的替代方案及否决理由 |
| 4 | 有风险评估吗？ | 列出设计中的技术风险和缓解措施 |

**参考 Google Design Doc**：设计文档必须包含目标、方案、替代方案、风险四个核心部分。

### 第二步：架构质量检查

| # | 检查项 | 通过标准 | 严重级别 |
|---|--------|---------|---------|
| 1 | 模块职责单一 | 每个模块职责描述 ≤ 3 句话，且不与其他模块重叠 | 阻塞 |
| 2 | 无循环依赖 | 模块间依赖关系是有向无环图 | 阻塞 |
| 3 | 接口定义完整 | 核心接口有输入输出字段、类型和校验规则 | 阻塞 |
| 4 | 数据流清晰 | 数据从输入到存储到输出的路径可端到端追溯 | 关键 |
| 5 | 错误处理覆盖 | 关键路径有明确的错误处理和降级策略 | 关键 |
| 6 | 可扩展性 | 新增功能不需要修改核心架构 | 一般 |
| 7 | 可测试性 | 每个模块可以独立测试（有明确接口） | 一般 |

**严重级别说明**：
- **阻塞**：必须修改才能通过评审
- **关键**：强烈建议修改，可在评审中标记为"有条件通过"
- **一般**：建议修改，不阻塞通过

### 第三步：非功能需求映射检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 性能需求有对应设计措施 | 每条性能需求在架构中有具体的支撑设计（缓存、异步、索引等） |
| 2 | 安全需求有对应设计措施 | 涉及敏感数据的模块有安全设计（加密、权限、审计） |
| 3 | 可用性需求有对应设计措施 | SLA 要求在架构中有具体支撑（冗余、降级、熔断等） |
| 4 | 可维护性有考量 | 架构支持独立部署、独立测试、独立变更 |

**输出**：非功能需求映射表

| 非功能需求 | 架构支撑措施 | 满足程度 | 遗留风险 |
|-----------|-------------|---------|---------|
| （示例）响应时间 < 200ms | Redis 缓存 + 异步处理 | 满足 | 高并发下需压测验证 |

### 第四步：蓝军挑战（可选，strict profile 必选）

参考华为蓝军机制，从反对者视角挑战设计：

| # | 挑战问题 | 目的 |
|---|---------|------|
| 1 | 如果用户量增长 10 倍，架构会怎样？ | 验证可扩展性假设 |
| 2 | 如果核心依赖（数据库/第三方服务）挂了，系统会怎样？ | 验证容错设计 |
| 3 | 如果这个模块需要完全重写，影响范围有多大？ | 验证模块独立性 |
| 4 | 有没有过度设计？哪些模块可以更简单？ | 防止过度工程化 |

**strict profile 要求**：蓝军必须提出至少 3 个"如果...会怎样"的挑战问题，每个问题有设计方的回应记录。

### 第五步：独立评审人（Bar Raiser）— 强制

参考 **Amazon Bar Raiser** 机制，技术评审必须引入至少 1 位项目外部的独立评审人。

**独立评审人资格要求**：
- 非当前项目组成员（不在 plan-tracker 的 DRI 列表中）
- 具备对应领域知识（能理解架构决策的技术含义）
- 无项目进度压力（不因"赶时间"降低标准）

**独立评审人职责**：
- 独立审查设计文档，不受项目组内部意见影响
- 有权对评审结论行使**否决权**——如果标准不达标，可以单方面阻止 Gate 通过
- 否决必须附带具体的否决理由和解除条件（什么情况下可以撤销否决）

**单 agent 场景下的最低标准**：
- Agent 必须从"对立面分析框架"重新审查设计（切换视角：从"为什么这样设计是对的"→"这个设计最可能在哪里失败"）
- 至少挑战 3 个核心假设（如"假设 X 服务永远可用""假设数据量不超过 Y"）
- 独立评审结论必须在切换分析框架后得出——不得复用同一分析框架

### 第六步：评审结论

| 结论 | 条件 | 后续动作 |
|------|------|---------|
| 通过 | 无阻塞项，关键项全部解决，独立评审人无否决 | 进入下一阶段 |
| 有条件通过 | 无阻塞项，关键项有遗留，独立评审人无否决 | 遗留项记录到跟踪表，下阶段关闭 |
| 需要修改 | 有阻塞项，独立评审人无否决 | 修改后重新评审 |
| **否决（Block）** | 独立评审人行使否决权 | **Gate 阻塞**——必须解除否决（满足解除条件+重新评审）才能通过 Gate。否决不能因"赶时间"被绕过 |

## 输出

| 产出物 | 格式 |
|--------|------|
| 评审结论 | 通过 / 有条件通过 / 需要修改 / 否决（Block） |
| 问题清单 | 每条有严重级别、描述、建议修改方案 |
| 非功能需求映射表 | 需求 → 措施 → 满足程度 |
| 遗留项列表（如有） | 每条有责任人和关闭截止日期 |
| 否决记录（如有） | 否决理由 + 解除条件 + 独立评审人 + 重评日期 |

## 质量标准

- [ ] 结构完整性 4 项检查全部通过
- [ ] 无阻塞级别的架构质量问题
- [ ] 非功能需求映射表完整（每条需求有对应措施）
- [ ] 独立评审人已指定且满足资格要求（非作者、非同一 agent、具备领域知识）
- [ ] 评审结论有明确理由；如有否决，否决理由和解除条件已记录

## 独立使用说明

本 skill 可脱离完整生命周期独立使用。

### 如果项目已初始化 governance（`.governance/` 存在）
1. 先执行目标锚定检查（读 plan-tracker → 确认成功标准 → 偏离检查）
2. 不需要先通过任何 Gate
3. 可用于任何设计文档的评审（不限于架构设计阶段）
4. 如果后续接入完整工作流，本 skill 的产出自动成为 G5（设计完成）的证据

### 如果项目未初始化 governance（`.governance/` 不存在）
1. 跳过所有治理前置检查
2. 直接执行本 skill 的核心流程（设计文档审查 + 蓝军挑战 + 评审结论）
3. 执行完毕后提醒用户："当前项目未初始化 governance。如需自动证据记录、Gate 检查、风险跟踪，运行 `/governance-init`。"

## 子工作流映射

| 本 skill 步骤 | 对应子工作流活动 | 映射说明 |
|--------------|-----------------|---------|
| 结构完整性检查 | 架构设计→概要设计 | 检查概要设计产出物的完整性 |
| 架构质量检查 | 架构设计→系统架构设计 | 逐项验证模块划分和接口定义质量 |
| 非功能需求映射 | 架构设计→需求到设计的映射 | 验证每个非功能需求有架构层面的支撑 |
| 蓝军挑战 | 架构设计→技术评审 | 华为蓝军机制的标准化执行方式 |
| 评审结论 | 架构设计→技术评审 | 直接输出为 Gate G5 要求的评审结论证据 |
