# 技术选型与方案预研 — 子工作流

本文件定义"技术选型与方案预研"阶段的独立子工作流。用户可单独加载本文件执行技术选型阶段。

## 阶段目标

从候选技术方案中选出最优方案。技术选型阶段的核心产出是"选型评估表"——回答"选什么、为什么、备选方案有什么、风险验证了吗"。

## 独立使用时的目标锚定

> 当本子工作流被独立加载（非通过 `main-workflow.md` 总入口加载）时，**MUST** 在执行阶段活动前完成：

1. **目标确认**：读取 `.governance/plan-tracker.md`，确认项目立项时定义的量化成功标准
2. **偏离检查**：对照成功标准，确认选型方向与项目总目标一致。如发现偏离，**MUST** 先记录到 `.governance/decision-log.md` 再继续
3. **质量底线**：选型决策必须有方案对比和"为什么不选其他"的理由（参考 profile 质量要求）

## 进入条件

- 调研阶段已完成（G2 通过）
- 有明确的调研结论和竞争分析支撑

## 活动清单

### 1. 选型范围确认

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 列出待选型的技术领域 | 框架、语言、数据库、部署方案、第三方服务等 | 需用户确认：哪些领域需要正式选型 |
| 确认选型优先级 | 核心架构选型优先，周边工具可快速决策 | 需用户确认 |

### 2. 候选方案列举

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 搜索候选方案 | 每个领域至少列举 2 个候选 | 自动：基于调研结论和公开资料 |
| 建立评估维度 | 性能、生态、学习曲线、团队熟悉度、许可、维护活跃度等 | 自动：基于项目约束推荐维度，需用户确认 |
| 填充评估矩阵 | 逐候选按维度评估 | 自动 |

### 3. PoC 验证

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 确定验证场景 | 选型中最关键的不确定性需要 PoC 验证 | 需用户确认：验证哪些关键假设 |
| 执行 PoC | 搭建最小验证环境 | 自动 |
| 记录 PoC 结果 | 结论为"验证通过"、"风险可控"或"不推荐" | 自动 |

### 4. 决策记录

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 输出选型结论 | 推荐方案 + 选择原因 + 排除理由 | 自动 |
| 记录到决策日志 | 决策编号、选择原因、备选方案 | 自动 |
| 用户确认 | 确认选型结论 | 需用户确认 |

## 产出物标准

| 产出物 | 格式要求 | 质量标准 |
|--------|---------|---------|
| 选型评估表 | ≥2 候选 × ≥4 维度，有评分或判定 | 矩阵完整无空白 |
| PoC 结果 | 关键假设的验证结论 | 结论明确（通过/风险可控/不推荐） |
| 决策记录 | 选择原因、备选方案、排除理由 | 决策可追溯到评估矩阵 |

## 退出条件

- [ ] 至少评估了 2 个候选方案
- [ ] 评估标准在评估前已定义
- [ ] 选择原因已留痕（含备选方案和排除理由）
- [ ] 关键风险已通过 PoC 验证

## Gate 映射

本阶段对应 **G3 — 选型完成**。

| Gate 检查项 | 对应活动 | 判定标准 |
|------------|---------|---------|
| 是否评估了至少 2 个候选方案？ | 候选方案列举 | 选型评估表列出 ≥2 个候选 |
| 评估标准是否事先定义？ | 候选方案列举 | 评估标准在评估前已定义 |
| 选择原因是否已留痕？ | 决策记录 | 决策记录有选择原因、备选方案和排除理由 |
| 关键风险是否已通过 PoC 验证？ | PoC 验证 | PoC 结果文档存在，结论为"验证通过"或"风险可控" |
