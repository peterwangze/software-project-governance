# Stage Skill 深度评分 Rubric

本文件定义 11 个 stage skill（`skills/stage-*/SKILL.md`）的统一深度评分标准。用于：
- 对新 stage skill 做自检：是否达到了最低深度门槛
- 对现有 stage skill 做均衡审计：哪些 skill 需要补强
- 对 Developer agent 提供明确的改进目标："从 21 分提升到 24 分"意味着什么

## 评分维度

6 个维度，每个 1~5 分，满分 30 分。

### D1. 活动组密度 (Activity Group Richness)

衡量 `## 活动清单` 下独立活动组的数量和结构层次。活动组定义为 `### N.` 标题下的内容块。

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 1~2 个活动组，缺少核心活动 | 只有"实现"和"检查"两个组 |
| 2 | 3 个活动组 | — |
| **3** | **4 个活动组**（当前 11 个 skill 的最低值） | operations: 数据收集/问题分类/运营分析/运营报告 |
| 4 | 5 个活动组 | testing: 测试计划确认/功能测试/非功能测试/测试报告/缺陷管理 |
| 5 | 6 个活动组，或 5 个活动组 + 有子节（如 3.1/3.2/3.3/3.4） | maintenance: 6 组；release: 5 组 + 3.1~3.4 子节 |

**自动化检查**：`grep -c '^### [0-9]' SKILL.md` 统计活动组数。子节检测：`grep -c '^#### [0-9]' SKILL.md`。

### D2. AI 编程风险覆盖 (AI Risk Coverage)

衡量 `## AI 编程场景特有风险` 表格中风险条目数量和缓解方向的具体性。

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 0~2 条风险，缓解方向模糊（如"注意检查"） | — |
| 2 | 3 条风险 | — |
| **3** | **4 条风险**（当前 11 个 skill 的最低值） | initiation: 4 条 |
| 4 | 5 条风险，缓解方向有具体动作 | testing: 5 条，每条缓解方向有具体名称的检查项 |
| 5 | 6 条风险，全部缓解方向具体、可执行、可检查 | development: 6 条（代码质量波动/跳过测试/虚假实现/漂移累积/过度实现/不安全代码） |

**自动化检查**：`grep -c '^| \*\*.*\*\* |' SKILL.md` 在 `## AI 编程场景特有风险` 和下一个 `##` 之间统计风险行。缓解方向具体性需人工判断。

### D3. Gate 自动化判定率 (Gate Automation Ratio)

衡量 `## Gate 映射` 表格中 Gate 检查项可被脚本/工具自动判定的比例。

| 分数 | 标准 |
|:---:|------|
| 1 | < 40% 的 Gate 项标记为"可自动" |
| 2 | 40% ~ 54% |
| **3** | **55% ~ 69%** |
| 4 | 70% ~ 84% |
| 5 | >= 85% 的 Gate 项标记为"可自动" |

**自动化检查**：
```bash
# 统计 Gate 映射表中标记为"可自动"的行数占比
total=$(grep -c '^| ' SKILL.md |  # Gate 检查项行
auto=$(grep -c '\*\*可自动\*\*' SKILL.md)
ratio=$((auto * 100 / total))
```

注意："半自动"或"需人工判断"不计数。只有明确标注"**可自动**"的才计入分子。

### D4. 企业实践溯源深度 (Enterprise Practice Mapping Depth)

衡量 `## 企业实践溯源` 表格中映射的企业实践数量，以及映射到具体活动的精确度。

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 0~3 条实践 | — |
| 2 | 4~5 条实践 | — |
| **3** | **6~7 条实践**，来自 >= 2 家企业 | initiation: 7 条（Google/Amazon/华为/字节） |
| 4 | 8~9 条实践 | release: 8 条（Google/Google SRE/Amazon/华为/字节） |
| 5 | >= 10 条实践，或包含"AI 编程专项"行 | testing: 11 条（含 AI 专项）；maintenance: 11 条 |

**自动化检查**：`grep -c '^| \*\*' SKILL.md` 在 `## 企业实践溯源` 表格中统计企业行数。企业多样性需人工判断。

### D5. 降级行为完整性 (Degradation Behavior Completeness)

衡量 `## 独立使用时的目标锚定` 节的质量：锚定步骤的具体程度、降级行为的完整性和可操作性。

| 分数 | 标准 |
|:---:|------|
| 1 | 无降级行为说明，或仅一句话（如"降级不可用"） |
| 2 | 提及降级但不具体（如"告知用户"但没有告知内容和判断逻辑） |
| **3** | **有 3 步锚定（目标确认/偏离检查/质量底线），降级有基本说明**（如 plan-tracker 不存在时内联告知） |
| 4 | 3 步锚定包含具体文件路径（如"读取 .governance/plan-tracker.md → 找到 ## 项目配置 节"），降级行为有具体通知语言和用户选项 |
| 5 | 3 步锚定有文件路径 + 降级覆盖多种场景（governance 缺失、目标缺失、profile 不匹配）+ 有"后续恢复"说明 + 通知语言区分"提问"和"单向通知" |

**自动化检查**：`grep -c '降级' SKILL.md` 检查降级相关段落数量。锚定步骤的完整性需人工判断。

### D6. 深度增强内容 (Depth-Enhancing Content)

衡量是否包含超出 7 个基本节（目标/风险/锚定/进入条件/活动/产出物/退出条件/Gate/溯源）的深度内容——如策略对比表、决策框架、专项检查清单、项目特定适配节等。

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 仅 4 个活动组 + 基本节，无任何增强内容 | — |
| 2 | 活动描述有基本具体性，但无独立增强节 | operations: 有详尽的 AI 风险描述但无增强节 |
| **3** | **1 个增强节** — 如活动内的子节、专项检查清单、或比较矩阵 | selection: PoC 验证节含置信度标注；infra: 环境可用性验证策略 |
| 4 | **2 个增强节** — 如专项检查清单 + 决策框架，或活动子节 + 项目适配节 | development: AI 代码专项检查 + 过度实现检查；maintenance: 技术债务盘点 + Bug 回归测试 |
| 5 | **>= 3 个增强节** — 含策略比较表、决策矩阵、专项管理框架等 | release: 发布策略表(5 策略) + Feature Flag 管理 + Kill Switch 验证 + 影响评估含无数据替代表；testing: PR/FAQ 验证 + AI 安全专项 + 缺陷根因分析 + 本项目测试类型对应表 |

**自动化检查**：统计 `###` 下嵌套的 `####` 子节数 + 非标准节的 `###` 标题数（注意排除：阶段目标/AI风险/锚定/进入条件/活动清单/产出物标准/退出条件/Gate映射/企业实践溯源）。

## 最低门槛

每个 stage skill 必须满足以下最低标准，不满足 = 不合格。

| 维度 | 最低分 | 说明 |
|------|:---:|------|
| D1 活动组密度 | >= 3 | 至少 4 个活动组 |
| D2 AI 风险覆盖 | >= 3 | 至少 4 条 AI 风险 |
| D3 Gate 自动化率 | >= 3 | 至少 55% 的 Gate 项可自动判定 |
| D4 企业实践映射 | >= 3 | 至少 6 条企业实践 |
| D5 降级行为完整性 | >= 3 | 至少有基本的 3 步锚定 + 降级说明 |
| D6 深度增强内容 | >= 3 | 至少有 1 个增强节 |
| **总分** | **>= 18** | 每维最低分 3 × 6 = 18 |

任一个维度低于最低分 = **Block**，必须修复该维度才能通过。

## 均衡目标

"均衡"指各维度深度比例合理（没有明显短板），且总分达到一定水准。均衡是目标，非最低门槛。

| 指标 | 目标值 | 说明 |
|------|:---:|------|
| 总分 | >= 22 | 平均 >= 3.67/5 分 |
| D1~D4 | 各 >= 4 | 结构维度（活动/风险/Gate/溯源）——量化基础必须扎实 |
| D5 | >= 3 | 降级行为——当前全 11 个 skill 均为 3 分，暂无 4+ 标杆 |
| D6 | >= 3 | 增强内容——取决于阶段业务复杂度，不强求 4+ |

**达到均衡目标的含义**：D1~D4 量化维度达到 Tier 1 标准，D5/D6 至少达到最低门槛。总分的 6 分差距（18 min -> 22 target）应优先投入到 D1~D4。

## 自动化检查

以下是可脚本化的检查项列表。每个检查项标注了对应的评分维度。

### 结构存在性检查（全部 7 个必需节）

```bash
# 每个 SKILL.md MUST 包含的 7 个节标题
required_sections=(
  "## 阶段目标"
  "## AI 编程场景特有风险"
  "## 独立使用时的目标锚定"
  "## 进入条件"
  "## 活动清单"
  "## 产出物标准"
  "## 退出条件"
  "## Gate 映射"
  "## 企业实践溯源"
)
# 检查每个节是否存在：grep -c 每个标题
```

### D1: 活动组计数

```bash
# 统计活动组数（### N. 标题）
activity_groups=$(grep -c '^### [0-9]\.' SKILL.md)
# >= 4 组 = 通过最低门槛（>= 3 分）

# 统计深度子节（#### N.N 标题）
sub_sections=$(grep -c '^#### [0-9]\.' SKILL.md)
# > 0 = D1 可额外加分
```

### D2: AI 风险数

```bash
# 在 "AI 编程场景特有风险" 和下一个 "## " 之间统计风险行
# 风险行的模式：| **风险名** | 说明 | 缓解方向 |
risk_count=$(sed -n '/## AI 编程场景特有风险/,/^## /p' SKILL.md | grep -c '^| \*\*')
# >= 4 条 = 通过最低门槛（>= 3 分）
```

### D3: Gate 自动化率

```bash
# 统计 Gate 映射表中标记为"可自动"的行数占比
total_gate=$(sed -n '/## Gate 映射/,/^## /p' SKILL.md | grep -c '^| .*|.*|.*|')
auto_gate=$(sed -n '/## Gate 映射/,/^## /p' SKILL.md | grep -c '\*\*可自动\*\*')
# 注意：排除表头行和分隔行
# ratio = auto / total >= 0.55 = 通过最低门槛（>= 3 分）
```

### D4: 企业实践数

```bash
# 在企业实践溯源表格中统计企业行
practice_count=$(sed -n '/## 企业实践溯源/,/^---$/' SKILL.md | grep -c '^| \*\*')
# >= 6 条 = 通过最低门槛（>= 3 分）
```

### D5: 降级行为存在性

```bash
# 检查降级关键词出现次数
degradation_mentions=$(grep -c '降级' SKILL.md)
# > 0 = 有降级处理（>= 2 分）；>= 3 且含 "plan-tracker" = >= 3 分

# 检查是否包含 3 步锚定
has_goal_confirm=$(grep -c '目标确认' SKILL.md)
has_deviation=$(grep -c '偏离检查' SKILL.md)
has_quality=$(grep -c '质量底线' SKILL.md)
# 三者都有 = D5 >= 3 分
```

### D6: 深度增强内容检测

```bash
# 统计非标准节的 ### 标题（排除标准节标题）
standard_sections='阶段目标|AI 编程场景特有风险|独立使用时的目标锚定|进入条件|活动清单|产出物标准|退出条件|Gate 映射|企业实践溯源'
extra_sections=$(grep '^### ' SKILL.md | grep -vE "$standard_sections" | wc -l)

# 统计 #### 子节
sub_sections=$(grep -c '^#### ' SKILL.md)

# 检测特殊表格关键词
has_strategy_table=$(grep -c '策略.*适用场景' SKILL.md)
has_comparison_matrix=$(grep -c '对比.*维度' SKILL.md)
has_fallback=$(grep -c '替代.*判定' SKILL.md)
# extra_sections + sub_sections + special_tables >= 1 = 通过最低门槛（>= 3 分）
```

## 使用指南

### Developer agent 使用本 rubric 做 stage skill 改进

1. 运行自动化检查脚本获取每个维度的原始数据
2. 对照本 rubric 的评分标准打出当前分数
3. 查看"当前 11 Stage 评分"表，定位目标分数
4. 查看"改进优先级"表，确定最小改动路径
5. 改进后重新评分，确认达到目标分数

### Reviewer agent 使用本 rubric 做 stage skill 审查

1. 对照 D1~D6 逐项评分
2. 任一维度 < 3 = Block（退回修改）
3. 总分 < 18 = Block
4. 总分在 18~21 = 条件通过（标注低分维度改进建议）
5. 总分 >= 22 = 通过

### Coordinator 使用本 rubric 做 stage skill 质量门控

- 新 stage skill 提交前 MUST 通过最低门槛检查（>= 18 分，每维 >= 3）
- 现有 stage skill 修改不得降低任何维度得分（回归检查）
- 每季度对所有 11 个 stage skill 重新评分，更新均衡状态
