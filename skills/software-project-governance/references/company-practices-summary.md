## Google — 设计评审与代码质量

### Design Doc（设计文档）

**什么时候用**：任何非平凡功能实现前、架构决策、跨模块变更。触发信号——用户说"我们要做 X"但还没写方案。

**可执行检查项**：
- [ ] 设计文档包含 5 个必需章节：目标、方案、替代方案（≥2 个）、风险、非功能需求覆盖。缺任何一章 = Design Doc 不合格
- [ ] 替代方案章节每个方案有"为什么不选"的具体理由。只列方案不写理由 = 不合格
- [ ] 文档控制在 1~3 页（或等量叙述）。超过 = 太复杂，需要拆分

**适用 profile**：strict（强制）> standard（强制）> lightweight（核心模块建议）

### Code Review（代码审查）

**什么时候用**：每一次代码变更合并前。触发信号——PR 被创建、commit 准备合并。

**可执行检查项**：
- [ ] 每条 Review 意见有回复：采纳（做了什么修改）或拒绝（为什么不采纳）。遗留未回复意见 = 阻塞合并
- [ ] 核心模块（数据模型/API/安全相关）有 ≥2 位 reviewer。AI 生成的代码同样需要逐行审查——不能因为是 AI 写的降低标准
- [ ] Review 不只是看"代码对不对"，更要看"设计是否合理"——是否存在过度工程、不必要的抽象、偏离架构意图

**适用 profile**：strict（核心 ≥2 reviewer）> standard（≥1 reviewer）> lightweight（关键路径 ≥1 reviewer）

### OKR 对齐（目标定义）

**什么时候用**：立项阶段定义项目目标时、每轮迭代开始前。触发信号——用户说"我们的目标是..."但没有量化。

**可执行检查项**：
- [ ] 目标 (Objective) 用一句话说清方向。不要"做一个好的系统"——要说"让用户能在 X 秒内完成 Y"
- [ ] 每个 KR (Key Result) 有数字：基线值 + 目标值 + 截止日期。"提升性能"不是 KR，"P99 延迟从 200ms 降到 100ms"才是
- [ ] 目标能向上追溯到上级 OKR，或明确声明"独立目标"。独立目标是合法的——但不声明就不合法

**适用 profile**：strict / standard（强制量化 KR）> lightweight（至少 1 个量化指标）

### Blameless Postmortem（无追责复盘）

**什么时候用**：任何线上事故、严重缺陷、或项目阶段结束后。触发信号——"线上出了个问题""这次做得不太好"。

**可执行检查项**：
- [ ] 复盘文档包含：时间线（按分钟级还原事实）、根因分析（5-Why 至少 3 层）、改进项（每条有责任人和截止日期）
- [ ] 复盘不追责个人，聚焦"流程/机制/工具哪里出了问题"。如果原因写的是"XX 操作失误"而没有追问"为什么这个失误没有被拦截"——不合格
- [ ] 每次复盘产出 ≤5 条改进项。超过 = 没排序、没聚焦

**适用 profile**：strict / standard（每次事故强制执行）> lightweight（严重事故强制执行）

## Amazon — 逆向工作与深度文档

### Working Backwards (PR/FAQ)

**什么时候用**：立项阶段、任何新功能启动前。触发信号——用户说"我们要做一个..."但没从用户视角描述过。

**可执行检查项**：
- [ ] "新闻稿" (Press Release) 已写——用 1 页纸从用户视角描述项目成功后是什么样的。不是技术语言，是用户语言
- [ ] "常见问题" (FAQ) 已写——包含外部 FAQ（用户会问什么）和内部 FAQ（团队会担心什么）。外部 FAQ ≥3 条，内部 FAQ ≥3 条
- [ ] PR/FAQ 中承诺的用户价值在验收阶段逐条对照验证——不是写完了就忘了

**适用 profile**：standard / strict（强制 PR/FAQ）> lightweight（建议写新闻稿）

### 6-Pager / Narrative（结构化论证）

**什么时候用**：技术选型、架构决策、重大方案变更。触发信号——"我们在选 A 还是 B""这个架构有几种方案"。

**可执行检查项**：
- [ ] 文档包含 6 段式结构：背景 → 问题 → 方案对比 → 收益分析 → 风险评估 → 时间线。不接受"PPT 式散点"——必须是连贯叙述
- [ ] 结论先行——读者读完第一段就知道推荐方案是什么。不是层层铺垫最后揭秘
- [ ] 方案对比矩阵有量化维度（性能/成本/复杂度），不是"A 比 B 好"

**适用 profile**：strict（强制 6-Pager 格式）> standard（核心决策时使用）> lightweight（建议但非强制）

### Bias for Action（70% 决策）

**什么时候用**：每个决策点。触发信号——"我们需要更多信息才能决定""再调研一下"。

**可执行检查项**：
- [ ] 每个决策已标注可逆性：可逆（框架选择、工具切换）= 70% 信息即推进；不可逆（数据模型、核心 API）= 90%+ 信息再决策
- [ ] "不做什么"列表包含 ≥1 条"看起来应该做但决定不做"的事项 + 理由。没有 = 范围边界不够清晰
- [ ] 有没有为了"未来可能需要"而正在做的功能？有 = 立即停止，标记为"等需要时再说"

**适用 profile**：全 profile 通用（这是决策纪律，不是可选项）

### Bar Raiser（独立评审）

**什么时候用**：架构评审、重要技术决策、验收阶段。触发信号——"我们内部评审过了"。

**可执行检查项**：
- [ ] 评审引入 ≥1 位项目外部独立 reviewer（Bar Raiser）。外部 = 不在当前项目组、不受项目进度压力影响
- [ ] Bar Raiser 有明确的否决权——如果评审不通过，项目不能因为"时间紧"绕过
- [ ] 评审结论有书面记录：通过/有条件通过/需修改，以及具体条件和截止日期

**适用 profile**：strict（强制 Bar Raiser）> standard（核心决策时建议）> lightweight（可选）

## 华为 — 门控治理与自我批判

### IPD 门控（阶段门禁）

**什么时候用**：每个阶段结束时。触发信号——用户说"这个阶段差不多了"。

**可执行检查项**：
- [ ] 每个 Gate 的检查项逐项打分（strict profile：0~5 分量化；standard：通过/有条件通过/未通过；lightweight：通过/阻塞）
- [ ] 前序阶段的 Gate 未通过，不得进入下一阶段。"先做着，Gate 后补"= 流程违规
- [ ] 有条件通过 (passed-with-conditions) 必须有明确的完成条件和截止日期。没有截止日期的条件 = 永远不会完成的条件

**适用 profile**：strict（量化评分强制）、standard（通过/未通过/有条件）、lightweight（仅通过/阻塞）

### 蓝军机制（对立面审查）

**什么时候用**：技术选型、架构设计、重大风险决策。触发信号——"这个方案看起来不错""大家都同意"。

**可执行检查项**：
- [ ] 指定 ≥1 人担任"蓝军"，职责是挑战假设和找漏洞——不是当反对派，是做风险探测
- [ ] 蓝军提出 ≥3 个"如果...会怎样"的挑战问题，每条有记录和回应。如果 3 个都提不出来 = 你对方案思考不够深
- [ ] 蓝军挑战完成后有结论：原方案通过 / 修订后通过 / 方案不可行。不是"听了但没改变"——每条挑战必须改变方案或被明确驳回

**适用 profile**：strict（核心决策强制蓝军）> standard（架构/选型时建议）> lightweight（可选）

### 自我批判（复盘纪律）

**什么时候用**：每个阶段结束后、项目总结时。触发信号——"我们总结一下""这个阶段结束了"。

**可执行检查项**：
- [ ] "做得不好"至少 3 条，每条有具体事例。不得使用模糊词——"基本完成""大致达到""总体良好"全部打回重写
- [ ] 每条"做得不好"有对应改进计划——不是"下次注意"而是"下次做 X 来防止 Y"
- [ ] 改进计划有责任人和截止日期——"团队一起改进"= 没有人负责

**适用 profile**：全 profile 通用（"做得不好"≥3 条强制；lightweight 至少 1 条）

### 让听得见炮声的人呼唤炮火（一线自主）

**什么时候用**：开发实现阶段、运营阶段。触发信号——实现过程中需要偏离设计、一线人员发现问题。

**可执行检查项**：
- [ ] 开发人员在不改变架构意图的前提下，可以自主调整实现方案。调整后必须记录到决策日志（含：改了什么/为什么/影响评估）
- [ ] 运营阶段发现的问题 48 小时内记录到问题清单并分配处理人。超过 48 小时未响应 = 流程违规
- [ ] 阻塞/关键级问题有 5-Why 根因分析。只修表象不查根因 = 问题会再来

**适用 profile**：全 profile 通用

## 字节跳动 — 数据驱动与上下文管理

### OKR 全员公开（透明度）

**什么时候用**：立项阶段、每次目标调整时。触发信号——"我们的目标是什么"。

**可执行检查项**：
- [ ] 项目目标、范围、进度对项目所有成员可见。对应本工作流的"单一事实源"原则——所有治理文件在 `.governance/` 目录下，全项目共享
- [ ] Key Results 有基线数据和目标数据——"提升用户活跃度"不合格，"DAU 从 10,000 提升到 15,000"合格
- [ ] 目标向上追溯路径清晰——本项目的 KR 对应哪个上级 O？如果独立，是否已声明？

**适用 profile**：strict / standard（强制公开 OKR）> lightweight（建议）

### A/B Test 验证（数据决策）

**什么时候用**：功能上线前、运营优化时。触发信号——"上个版本改了什么""改完好还是不好"。

**可执行检查项**：
- [ ] 上线前后数据有对比——性能基线 vs 当前、错误率基线 vs 当前、用户活跃度基线 vs 当前。没有基线 = 没有结论
- [ ] 决策用数据支撑而非主观判断。"感觉变快了"= 不合格，"P99 延迟从 500ms 降到 200ms"= 合格
- [ ] 测试报告必须有量化数据——通过率/覆盖率/性能数据。不接受"测试通过了"

**适用 profile**：standard / strict（强制数据对比）> lightweight（关键指标建议对比）

### Context not Control（上下文授权）

**什么时候用**：任务分配、方案交接时。触发信号——"你去做 X""这块你负责"。

**可执行检查项**：
- [ ] 分配任务时传递"为什么做"而不只是"做什么"——让执行者理解决策上下文
- [ ] 架构文档/ADR 记录"为什么这样设计"而不只是"设计成什么样"——后来的维护者需要理解原因
- [ ] 团队成员理解上下文后，在边界内自主决策。不需要每件事都"等领导确认"

**适用 profile**：全 profile 通用

### 坦诚清晰（信息最短路径）

**什么时候用**：日常协作中。触发信号——"私下沟通一下""会议后再对齐"。

**可执行检查项**：
- [ ] 信息用最短路径传递——数据直接给原始文件而非经过 PPT 二次过滤
- [ ] 存在单一事实源——同一信息不重复存储在多个地方。如果有两份文件描述同一件事 = 未来必然不一致
- [ ] 反馈直接给当事人而非"先跟别人说再转达"——缩短信息传递链

**适用 profile**：全 profile 通用

## 实践速查卡

以下是各主要场景的"最少必做项"清单——如果只能做 3 件事，就做这些：

| 场景 | 最少必做项（3 条） | 来源 |
|------|------------------|------|
| **立项目标定义** | ① 量化成功标准（有数字）② 写用户视角"新闻稿" ③ 列出"不做什么"≥1 条 | Google OKR + Amazon PR/FAQ |
| **技术选型** | ① ≥2 个候选对比 ② 蓝军挑战 ≥3 个问题 ③ 标注可逆/不可逆 | Amazon 6-Pager + 华为蓝军 + Amazon Bias |
| **架构设计** | ① Design Doc 5 章齐全 ② 蓝军挑战完成 ③ Bar Raiser 独立评审 | Google Design Doc + 华为蓝军 + Amazon Bar Raiser |
| **开发实现** | ① TDD 顺序（先测试再实现）② Code Review 意见闭环 ③ AI 专项检查（无 mock 残留/硬编码） | Google Code Review + AI 专项 |
| **测试验证** | ① 阻塞/关键缺陷 = 0 ② 测试报告有量化数字 ③ 阻塞级缺陷 5-Why 分析 | 字节数据驱动 + 华为 RCA |
| **版本发布** | ① 发布 checklist 逐项通过 ② 回滚方案在测试环境跑过 ③ 发布后冒烟测试通过 | 华为 IPD + Amazon 回滚 + Google SRE |
| **复盘改进** | ① "做得不好"≥3 条具体事例 ② 根因分析 5-Why ≥3 层 ③ 改进项 ≤5 条有负责人 | 华为自我批判 + Google Blameless |
| **运营回收** | ① 数据 vs 基线对比 ② PR/FAQ 达成验证 ③ 问题清单有严重级别和处理人 | 字节 A/B + Amazon PR/FAQ + 华为 48h |

## Profile 适配指南

不同 profile 对"哪些检查是强制项"有不同要求。以下是分 profile 的最小强制集：

| 实践 | lightweight | standard | strict |
|------|:----------:|:--------:|:------:|
| 目标量化（数字指标） | ≥1 个量化指标 | ≥2 个量化 KR | ≥3 个量化 KR + 基线 |
| Design Doc 五段式 | 建议 | 强制 | 强制 + Bar Raiser |
| Code Review | ≥1 reviewer | ≥1 reviewer + 意见闭环 | ≥2 reviewer（核心模块）|
| 蓝军挑战 | 可选 | 建议（架构/选型） | 强制（架构/选型/重大决策） |
| Bar Raiser | 可选 | 建议（重大决策） | 强制 |
| Gate 量化评分 | 通过/阻塞 | 通过/有条件/未通过 | 0-5 分量化 |
| 复盘"做得不好" | ≥1 条 | ≥3 条 | ≥3 条 + 5-Why 根因 |
| 数据基线对比 | 建议 | 强制 | 强制 + 统计显著性 |
