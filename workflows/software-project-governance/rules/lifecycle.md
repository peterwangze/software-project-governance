# 生命周期规则

本文件定义 `software-project-governance` workflow 的三层阶段模型、触发模式和项目治理规则。

## 三层架构

本工作流采用三层内容架构：

1. **项目整体治理层** — 覆盖全流程的目标看护、阶段推进节奏、全局 Gate 把控
2. **阶段子工作流层** — 每个阶段有独立的进入条件、活动清单、产出物标准和退出条件
3. **具体事务 skill/script 层** — 每个阶段内部的具体操作工具

agent 可以按需加载任意层级：
- 加载整体治理层 → 获得全流程看护能力
- 只加载某个阶段的子工作流 → 获得该阶段的独立执行能力
- 只加载某个 skill → 获得该具体事务的执行能力

## 阶段列表

本工作流定义以下阶段。每个阶段在 `stages/<stage-id>/sub-workflow.md` 中有独立的子工作流定义。

### 1. 立项与目标定义（initiation）

- **目标**：明确问题、目标、边界、成功标准和关键约束
- **核心活动**：问题陈述、目标定义、范围边界、成功标准、干系人识别、初始风险评估
- **产出物**：项目章程、目标说明、范围边界文档、初始风险清单
- **退出条件**：目标可衡量、范围有边界、干系人已识别、初始风险已记录
- **子工作流**：`stages/initiation/sub-workflow.md`

### 2. 调研与竞争分析（research）

- **目标**：通过调研和竞争分析建立决策事实基础
- **核心活动**：市场/技术调研、竞争产品分析、用户需求分析、可行性评估
- **产出物**：调研报告、竞争分析矩阵、可行性评估、关键发现总结
- **退出条件**：调研覆盖关键决策维度、竞争格局清晰、可行性结论明确
- **子工作流**：`stages/research/sub-workflow.md`

### 3. 技术选型与方案预研（selection）

- **目标**：基于调研结果做出技术选型和方案方向决策
- **核心活动**：候选方案列举、评估标准定义、PoC/原型验证、方案对比、决策留痕
- **产出物**：选型评估表、PoC 结果、技术方案决策记录、风险评估更新
- **退出条件**：技术方向已选定、关键风险已验证、决策已记录
- **子工作流**：`stages/selection/sub-workflow.md`

### 4. 环境搭建与基础设施（infrastructure）

- **目标**：建立项目开发、测试、部署的基础设施
- **核心活动**：开发环境配置、代码仓库初始化、CI/CD 基础搭建、工具链选型、权限与规范
- **产出物**：环境配置文档、仓库结构、CI/CD 基础配置、工具链说明
- **退出条件**：开发环境可用、仓库结构已建立、基础 CI 可运行
- **子工作流**：`stages/infrastructure/sub-workflow.md`

### 5. 架构设计（architecture）

- **目标**：形成可执行的技术方案和系统设计
- **核心活动**：系统架构设计、模块划分、接口定义、概要设计、详细设计、技术评审
- **产出物**：架构设计文档、模块划分图、接口定义、概设/详设文档、技术评审结论
- **退出条件**：架构评审通过、关键接口已定义、详细设计覆盖核心模块
- **子工作流**：`stages/architecture/sub-workflow.md`

### 6. 开发实现（development）

- **目标**：按设计产出可工作的软件资产
- **核心活动**：迭代开发、代码编写、单元测试、Code Review、集成调试
- **产出物**：源代码、单元测试、Code Review 记录、集成验证结果
- **退出条件**：核心功能已实现、单元测试覆盖达标、Code Review 遗留项已关闭
- **子工作流**：`stages/development/sub-workflow.md`

### 7. 测试与质量保障（testing）

- **目标**：验证软件质量达到发布标准
- **核心活动**：集成测试、系统测试、性能测试、安全测试、回归测试、缺陷管理
- **产出物**：测试计划、测试用例、测试报告、缺陷清单、质量评估
- **退出条件**：关键缺陷已关闭、回归测试通过、性能指标达标
- **子工作流**：`stages/testing/sub-workflow.md`

### 8. 防护网与 CI/CD（ci-cd）

- **目标**：建立持续集成、持续交付的质量防护网
- **核心活动**：CI 流水线完善、自动化测试集成、代码质量门禁、部署流水线、监控告警
- **产出物**：CI/CD 配置、自动化测试套件、质量门禁规则、部署文档
- **退出条件**：CI 流水线稳定、自动化测试覆盖核心路径、质量门禁生效
- **子工作流**：`stages/ci-cd/sub-workflow.md`

### 9. 版本发布（release）

- **目标**：按计划发布版本并确保发布过程可控
- **核心活动**：发布计划、变更清单、发布测试、灰度/金丝雀发布、回滚方案
- **产出物**：发布计划、变更日志、发布 checklist、回滚方案、发布验证结果
- **退出条件**：发布验证通过、监控指标正常、回滚方案已验证
- **子工作流**：`stages/release/sub-workflow.md`

### 10. 运营与反馈（operations）

- **目标**：跟踪实际运行状况、收集反馈、识别改进点
- **核心活动**：运营监控、用户反馈收集、问题归档、性能跟踪、优化建议
- **产出物**：运营报告、反馈汇总、问题清单、优化建议
- **退出条件**：首轮运营数据已收集、关键问题已归档、优化方向已明确
- **子工作流**：`stages/operations/sub-workflow.md`

### 11. 维护与演进（maintenance）

- **目标**：基于复盘和反馈持续修订项目资产
- **核心活动**：问题修复、规则修订、模板更新、版本规划、复盘会议
- **产出物**：修复记录、规则修订、模板更新、复盘结论、下轮规划
- **退出条件**：复盘完成、修订已回灌、下轮方向已明确
- **子工作流**：`stages/maintenance/sub-workflow.md`

## 阶段关系

### 默认推进顺序

阶段按编号 1→2→3→...→11 顺序推进。每个阶段完成后通过对应 Gate 检查，再进入下一阶段。

### 阶段可重叠

以下相邻阶段允许在满足条件后并行推进：
- 调研（2）与技术选型（3）可重叠：调研中期即可开始候选方案评估
- 环境搭建（4）与架构设计（5）可重叠：基础环境先行，详细设计可迭代
- 开发（6）与测试（7）可重叠：测试用例编写可与开发并行
- 防护网（8）与版本发布（9）可重叠：CI 稳定后即可准备发布流程

### 阶段可回退

当后续阶段发现前置阶段遗漏时，允许回退：
- 回退必须更新风险记录和决策记录
- 回退不意味着"推翻已完成"，而是"补齐缺失"
- 回退范围由项目 profile 决定（轻量模式允许快速回退，严格模式要求正式审批）

## 触发模式

工作流支持三种触发模式，可在项目 profile 或会话中配置。

### always-on（默认开启）

- 每次会话自动加载工作流
- 持续跟踪项目状态和阶段推进
- 所有治理动作自动执行（补证据、更新风险、检查 Gate）
- 适用场景：正式项目、团队协作项目

### on-demand（按需调用）

- 用户主动触发某个阶段或 skill 时才加载
- 不自动跟踪全局状态
- 只执行用户显式请求的治理动作
- 适用场景：已有成熟流程的项目、只想用某个 skill 的用户

### silent-track（静默跟踪）

- 后台记录项目变更，不打扰用户正常开发
- 只在 Gate 检查点或关键偏差时提醒用户
- 不主动要求用户填写记录，但会在后台生成建议
- 适用场景：个人项目、探索性项目、不希望被打扰的用户

### 模式切换

- 在项目 manifest 中声明默认触发模式
- 用户可在会话中临时切换模式
- 不同阶段可以配置不同的默认触发模式（如开发阶段用 silent-track，发布阶段用 always-on）

## 项目治理规则

### 目标看护

1. 项目的初始目标定义在立项阶段，记录在项目章程中
2. 后续任何阶段发生目标偏离，必须更新决策记录和风险记录
3. 每个阶段的退出条件必须包含"目标一致性检查"

### 阶段推进

1. 阶段推进前必须通过对应 Gate 检查（受 profile 调节）
2. 任意时刻至少有一个"当前阶段"（可重叠阶段除外）
3. 阶段状态变更必须回写统一事实源

### 过程可信

1. 已完成事项必须至少绑定一条证据
2. 关键取舍、范围变化、门禁变化必须进入决策记录
3. 阻塞项必须记录为风险，风险状态变化必须回写
4. Gate 结论必须能追溯到证据和事实源

### 统一要求

1. 不同 agent 不得维护不同版本的主计划
2. 所有治理记录集中写入统一事实源
3. 工作流规则变更必须经过决策记录留痕
4. 模板和规则可演进，但变更必须可追溯

## 与其他规则文件的关系

- `stage-gates.md`：定义每个阶段对应的 Gate 检查规则，支持裁剪和有条件通过
- `profiles.md`：定义轻量/标准/严格三种项目 profile，影响阶段范围和 Gate 强度
- `onboarding.md`：定义项目中途接入的协议和最小路径
- `stages/<stage-id>/sub-workflow.md`：定义每个阶段内部的子工作流
