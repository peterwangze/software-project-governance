# 用户视角强制原则

**工作流是给人用的，不是自嗨。** 所有规划、设计、开发、测试 MUST 站在用户视角回答三个问题。

## 三个必答问题

每项变更在设计阶段 MUST 回答：

| # | 问题 | 不合格的回答 | 合格的回答 |
|---|------|-----------|----------|
| 1 | **用户怎么获得这个变更？** | "改了代码" | "用户执行 `/plugin update` → `/reload-plugins` 后生效。新用户通过 governance-init 自动获得。" |
| 2 | **用户怎么知道这个变更存在？** | "写在 CHANGELOG 里了" | "CHANGELOG + `/governance-status` 会提示有新版本可用 + 首次使用时 agent 会告知新功能" |
| 3 | **用户的体验真的变了吗？** | "机制已经在了" | "已验证：用户初始化后 agent 第一次 commit 时 hook 触发，输出 governance 检查结果。用户可见。" |

## 用户视角检查清单

每项任务完成时 MUST 逐项检查：

- [ ] **可达性**：用户不需要手动搬文件、改配置、读源码就能获得这个变更。变更通过 `/plugin update` 或 governance-init 自动到达用户。
- [ ] **可知性**：用户不需要猜测这个功能是否存在。CHANGELOG、governance-status、或 agent 行为中会主动告知用户。
- [ ] **可验证性**：用户可以独立验证这个功能是否生效。不是"相信 agent 做了"——是"看到 agent 输出了什么"。
- [ ] **无断点**：从"用户当前状态"到"用户获得新功能"的路径上没有断点。如果用户需要手动做任何事，这个步骤有明确的文档和引导。
- [ ] **已安装用户不被遗忘**：变更不是只对新用户有效。已安装用户有明确的升级路径——不需要重新 init 就能获得改进。
- [ ] **默认行为改善**：用户不做任何配置就能感受到改善。新功能不要求用户学习新概念或记住新命令。

## 反模式（禁止）

以下行为被判定为"自嗨式开发"：

| 反模式 | 为什么是自嗨 | 用户实际体验 |
|--------|------------|------------|
| 只改 平台原生入口文件 狗粮实例 | 只在本仓库生效 | 用户安装了插件，agent 行为没变 |
| 只改 governance-init 模板 | 只对新用户有效 | 已安装用户的体验永远停在安装时的版本 |
| 只写文档不改机制 | 用户读完文档不知道怎么执行 | "知道了 Design Doc 很重要，但没人告诉我怎么写" |
| 版本号不 bump | 用户不知道有更新 | `/plugin` 说"已是最新"，实际内容已大变 |
| 功能没有触发信号 | 用户不知道功能存在 | "原来还可以这样？我从来不知道" |

## 应用范围

本原则应用于工作流自身的演进——**吃自己的狗粮**：

- **规划**：版本路线图中的每个版本 MUST 从用户视角描述交付价值（不是"修复了 X 机制"而是"用户现在可以 Y"）
- **设计**：每个机制变更 MUST 附带用户旅程描述（用户从 A 状态出发，经过 B 步骤，到达 C 体验）
- **开发**：每个代码变更 MUST 验证用户可达性（变更通过哪种方式到达用户？新用户/已安装用户分别怎么获得？）
- **测试**：每个验证 MUST 包含"用户真的能用吗"（不只是 verify_workflow.py PASSED，还要模拟用户路径）

## 集成到工作流

本原则已集成到以下治理机制：

- `平台原生入口文件` bootstrap：每次会话加载，agent 在干活前自动检查用户视角
- `SKILL.md` M1 触发条件：涉及用户可见变更时 MUST 加载本文件
- Gate 检查：G5（设计完成）和 G9（发布就绪）MUST 包含用户视角检查
- 审计框架 D5（易用性审计）：基于本文件的 6 项检查清单
