# 版本发布 — 子工作流

本文件定义"版本发布"阶段的独立子工作流。用户可单独加载本文件执行版本发布阶段。

## 阶段目标

将已验证的软件版本安全发布到目标环境。版本发布阶段的核心产出是"发布的版本"——用户可以使用的新版本，且有明确的变更记录和回滚能力。

## 独立使用时的目标锚定

> 当本子工作流被独立加载（非通过 `main-workflow.md` 总入口加载）时，**MUST** 在执行阶段活动前完成：

1. **目标确认**：读取 `.governance/plan-tracker.md`，确认项目立项时定义的量化成功标准
2. **偏离检查**：对照成功标准，确认本次发布内容与项目总目标一致。如发现偏离，**MUST** 先记录到 `.governance/decision-log.md` 再继续
3. **质量底线**：发布必须通过 checklist 全项检查且有可验证的回滚方案（参考 profile 质量要求）

## 进入条件

- 防护网已就绪（G8 通过）
- CI 稳定运行
- 有测试报告支撑发布决策

## 活动清单

### 1. 发布准备

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 确认发布范围 | 本次发布包含哪些功能 | 需用户确认 |
| 定义版本号 | 按语义化版本规则 | agent 执行 → 需用户确认 |
| 编写变更日志 | 覆盖所有变更 | agent 执行 |

### 2. 发布前检查

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 执行发布 checklist | 使用发布 skill 中的 5 步检查 | agent 执行 |
| 验证回滚方案 | 在测试环境执行回滚 | 需测试环境执行 |
| 确认监控就绪 | 关键指标监控和告警已配置 | 需监控环境执行 |

### 3. 发布执行

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 执行发布 | 按部署文档执行 | 需用户确认：发布决策 |
| 发布后验证 | 核心功能验证 | 需外部环境执行 |
| 监控观察 | 发布后指标是否正常 | 需监控环境执行 |

### 4. 发布收尾

| 活动 | 描述 | 交互边界 |
|------|------|---------|
| 更新版本记录 | 版本号、发布日期、变更范围 | agent 执行 |
| 通知相关人员 | 发布完成通知 | agent 执行 |
| 归档发布资产 | tag、changelog、发布包 | agent 执行 |

## 产出物标准

| 产出物 | 格式要求 | 质量标准 |
|--------|---------|---------|
| 发布计划 | 版本号、范围、时间窗口 | 完整且已确认 |
| 变更日志 | 覆盖所有变更，破坏性变更已高亮 | 与 commit 列表对照一致 |
| 回滚方案 | 具体步骤，已在测试环境验证 | 可执行 |
| 发布后验证报告 | 核心功能验证结果 | 全部通过 |

## 退出条件

- [ ] 发布范围明确（有版本号和变更清单）
- [ ] 变更日志覆盖所有变更
- [ ] 回滚方案已验证
- [ ] 发布后验证通过

## Gate 映射

本阶段对应 **G9 — 发布就绪**。

| Gate 检查项 | 对应活动 | 判定标准 |
|------------|---------|---------|
| 发布范围是否明确？ | 发布准备 | 有版本号、发布范围、时间窗口 |
| 变更日志是否完整？ | 变更日志 | 覆盖所有变更，与 commit 列表对照一致 |
| 回滚方案是否已验证？ | 发布前检查 | 有具体步骤且已在测试环境验证 |
| 发布后验证是否已定义？ | 发布前检查 | 验证 checklist 存在且包含核心功能 |
