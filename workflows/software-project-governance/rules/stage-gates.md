# Stage Gate 规则

本文件定义 `software-project-governance` workflow 的阶段门禁规则，支持 Gate 裁剪、有条件通过和中途接入。

## Gate 列表

每个 Gate 对应 `lifecycle.md` 中的一个阶段转换。

### G1 — 立项完成

- **阶段转换**：立项 → 调研
- **必审材料**：项目章程、目标说明、范围边界、初始风险清单
- **检查项**：
  1. 项目目标是否可衡量？
  2. 范围边界是否清晰？
  3. 关键干系人是否已识别？
  4. 是否有明确的"不做什么"清单？
- **通过标准**：目标可衡量、范围有边界、干系人已识别
- **有条件通过**：目标方向正确但量化指标暂缺 → 必须在调研阶段补齐
- **自动判定标准**：
  - 检查项1：成功标准包含至少 1 个量化指标（数字、百分比、时间约束）
  - 检查项2：范围外事项列表非空
  - 检查项3：干系人列表 ≥ 1 条，每条有角色描述
  - 检查项4：范围外事项列表有内容且与范围内不重叠

### G2 — 调研完成

- **阶段转换**：调研 → 技术选型
- **必审材料**：调研报告、竞争分析矩阵、关键发现总结
- **检查项**：
  1. 调研是否覆盖了技术、市场和用户三个维度？
  2. 竞争格局是否清晰？
  3. 关键发现是否有数据支撑？
  4. 是否识别了技术可行性的关键约束？
- **通过标准**：调研覆盖关键决策维度、竞争格局清晰、可行性有初步判断
- **自动判定标准**：
  - 检查项1：调研报告中有"技术"、"市场/竞品"、"用户/需求"三个独立章节
  - 检查项2：竞品对比矩阵 ≥ 3 个竞品 × ≥ 4 个维度，无空白项
  - 检查项3：每条关键发现标注了数据来源（URL、文档名、调研对象）
  - 检查项4：至少 1 条技术约束，包含影响描述和严重级别

### G3 — 选型完成

- **阶段转换**：技术选型 → 环境搭建
- **必审材料**：选型评估表、PoC 结果、技术方案决策记录
- **检查项**：
  1. 是否评估了至少 2 个候选方案？
  2. 评估标准是否事先定义？
  3. 选择原因是否已留痕？
  4. 关键风险是否已通过 PoC 验证？
- **通过标准**：技术方向明确、决策已记录、关键风险已验证
- **自动判定标准**：
  - 检查项1：选型评估表列出 ≥ 2 个候选方案
  - 检查项2：评估标准在评估前已定义（时间戳可验证）
  - 检查项3：决策记录中有选择原因、备选方案和排除理由
  - 检查项4：PoC 结果文档存在，且结论为"验证通过"或"风险可控"

### G4 — 环境就绪

- **阶段转换**：环境搭建 → 架构设计
- **必审材料**：环境配置文档、仓库结构、基础 CI 配置
- **检查项**：
  1. 开发环境是否可复现？
  2. 仓库结构是否符合约定？
  3. 基础 CI 是否可运行？
  4. 协作规范是否已建立？
- **通过标准**：环境可用、仓库已初始化、基础 CI 通过
- **自动判定标准**：
  - 检查项1：开发环境可在新机器上复现（有 Dockerfile/环境配置文档/一键脚本）
  - 检查项2：仓库目录结构与约定一致（验证脚本通过）
  - 检查项3：基础 CI 配置文件存在且 `ci run` 命令返回成功
  - 检查项4：协作规范文档存在（贡献指南/编码规范/提交规范至少 1 份）

### G5 — 设计完成

- **阶段转换**：架构设计 → 开发实现
- **必审材料**：架构文档、模块划分、接口定义、概设/详设、技术评审结论
- **检查项**：
  1. 架构是否满足非功能性需求？
  2. 模块划分是否清晰、职责是否单一？
  3. 关键接口是否已定义？
  4. 是否经过技术评审？
  5. 详细设计是否覆盖核心模块？
- **通过标准**：架构评审通过、核心接口已定义、详细设计覆盖主路径
- **有条件通过**：非核心模块详细设计暂缺 → 必须在开发阶段补齐
- **自动判定标准**：
  - 检查项1：架构文档中有非功能需求对应表（需求 → 设计措施）
  - 检查项2：模块列表中每个模块职责描述 ≤ 3 句话，模块间无循环依赖
  - 检查项3：核心接口有完整的输入输出字段定义和类型
  - 检查项4：有技术评审结论（通过/条件通过/需修改）
  - 检查项5：核心模块有内部结构描述（类图/数据结构/算法）

### G6 — 开发完成

- **阶段转换**：开发实现 → 测试
- **必审材料**：源代码、单元测试、Code Review 记录、集成验证结果
- **检查项**：
  1. 核心功能是否按设计实现？
  2. 单元测试覆盖率是否达标？
  3. Code Review 遗留项是否已关闭？
  4. 集成验证是否通过？
- **通过标准**：核心功能已实现、测试覆盖达标、Review 遗留项已关闭
- **自动判定标准**：
  - 检查项1：功能列表与设计文档逐项对照，无遗漏
  - 检查项2：coverage 报告 ≥ profile 阈值（lightweight: 无要求, standard: ≥70%, strict: ≥90%）
  - 检查项3：Review 工具中遗留项计数 = 0
  - 检查项4：集成测试套件全部通过，无阻塞性失败

### G7 — 测试通过

- **阶段转换**：测试 → 防护网与 CI/CD
- **必审材料**：测试计划、测试报告、缺陷清单、质量评估
- **检查项**：
  1. 关键缺陷是否已关闭？
  2. 回归测试是否通过？
  3. 性能指标是否达标？
  4. 安全测试是否覆盖关键风险？
- **通过标准**：关键缺陷关闭、回归通过、性能达标
- **自动判定标准**：
  - 检查项1：缺陷跟踪中关键/阻塞级别缺陷数 = 0
  - 检查项2：回归测试套件通过率 = 100%
  - 检查项3：性能测试结果在目标范围内（响应时间/吞吐量/资源占用）
  - 检查项4：安全测试覆盖 OWASP Top 10 关键项，无高危漏洞

### G8 — 防护网就绪

- **阶段转换**：防护网与 CI/CD → 版本发布
- **必审材料**：CI/CD 配置、自动化测试套件、质量门禁规则
- **检查项**：
  1. CI 流水线是否稳定运行？
  2. 自动化测试是否覆盖核心路径？
  3. 质量门禁是否生效？
  4. 部署流程是否文档化？
- **通过标准**：CI 稳定、自动化测试覆盖核心路径、门禁生效
- **自动判定标准**：
  - 检查项1：CI 最近 5 次运行成功率 ≥ 80%
  - 检查项2：自动化测试覆盖核心功能路径（路径列表与设计文档功能模块对应）
  - 检查项3：质量门禁规则文件存在且 CI 日志中可见门禁检查
  - 检查项4：部署文档存在且步骤可执行

### G9 — 发布就绪

- **阶段转换**：版本发布 → 运营
- **必审材料**：发布计划、变更日志、发布 checklist、回滚方案
- **检查项**：
  1. 发布范围是否明确？
  2. 变更日志是否完整？
  3. 回滚方案是否已验证？
  4. 发布后验证是否已定义？
- **通过标准**：发布计划完整、回滚方案可执行
- **自动判定标准**：
  - 检查项1：发布计划包含版本号、发布范围、时间窗口
  - 检查项2：变更日志覆盖本次所有变更（commit 列表与 changelog 对照）
  - 检查项3：回滚方案有具体步骤且已在测试环境验证过
  - 检查项4：发布后验证 checklist 存在且包含核心功能验证项

### G10 — 运营回收

- **阶段转换**：运营 → 维护
- **必审材料**：运营报告、反馈汇总、问题清单、优化建议
- **检查项**：
  1. 是否收集到真实运营数据？
  2. 用户反馈是否已归档？
  3. 关键问题是否已识别和分类？
  4. 优化方向是否已明确？
- **通过标准**：运营数据已收集、关键问题已归档
- **自动判定标准**：
  - 检查项1：运营报告包含至少 1 周的真实运行数据
  - 检查项2：用户反馈已归档（反馈条数 ≥ 0，有分类）
  - 检查项3：问题清单存在且每条有严重级别和状态
  - 检查项4：优化建议列表非空（至少 1 条可执行的改进方向）

### G11 — 维护闭环

- **阶段转换**：维护 → 下一轮立项或规划
- **必审材料**：复盘结论、规则修订、模板更新
- **检查项**：
  1. 是否完成复盘？
  2. 经验是否回灌到规则和模板？
  3. 下轮方向是否已明确？
  4. 版本化记录是否已更新？
- **通过标准**：复盘完成、修订已回灌、下轮方向明确
- **自动判定标准**：
  - 检查项1：复盘文档存在（包含目标回顾、结果评估、原因分析、经验沉淀）
  - 检查项2：规则/模板有对应的更新提交记录
  - 检查项3：下一轮方向在计划跟踪表中有 ≥ 1 条 P0 任务
  - 检查项4：manifest 或样例总览中的版本/日期字段已更新

## Gate 通过类型

### 通过（passed）

所有检查项满足，可以进入下一阶段。

### 有条件通过（passed-with-conditions）

核心检查项满足，但存在非阻塞性遗留项。条件：
- 遗留项必须记录在计划跟踪表中
- 遗留项必须在下一阶段结束前关闭
- 遗留项数量不超过 3 条

适用 profile：standard、strict（strict 下仅限明确允许有条件通过的 Gate）

### 未通过（blocked）

核心检查项不满足，不得进入下一阶段。

处理方式：
- 必须更新风险记录
- 必须制定纠偏动作
- 重新通过前需要再次检查

### 接入通过（passed-on-entry）

仅用于中途接入场景。前置阶段的 Gate 通过此状态承认"已完成"。

条件：
- 必须有对应的决策记录说明该阶段已完成
- 不要求补齐该阶段的全部证据
- 记录为"接入时承认通过"

## Gate 裁剪规则

### lightweight profile

- G1+G2 合并为一个"启动检查"
- G3、G4、G5 跳过
- G6+G7 合并为一个"质量检查"
- G8+G9 合并为一个"发布检查"
- G10、G11 仅在进入对应阶段时触发

### standard profile

- 全部 Gate 检查
- 支持有条件通过

### strict profile

- 全部 Gate 检查
- 不允许有条件通过
- 每个 Gate 需要至少两条独立证据

## Gate 执行原则

1. 未通过 Gate，不得声称进入下一阶段（受 profile 调节）
2. Gate 结论必须能回溯到证据和事实源
3. 任何 agent 适配都不得绕过 Gate 规则
4. Gate 检查可以由 agent 自动执行（可程序化部分）或由用户确认（需人工判断部分）
5. Gate 失败后的重新尝试不需要"回到上一阶段"，只需补齐缺失项后重新检查

## 与其他规则的关系

- `lifecycle.md`：阶段列表以 lifecycle 为准，Gate 编号与阶段对应
- `profiles.md`：profile 决定 Gate 的裁剪方式和通过类型
- `onboarding.md`：中途接入时前置 Gate 的处理方式
