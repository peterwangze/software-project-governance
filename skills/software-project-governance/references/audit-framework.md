# Audit Framework — 审计作为一等治理看护手段

本文件定义 `software-project-governance` 中的审计机制。审计不是一个额外阶段，而是贯穿所有阶段和所有 Gate 的**交叉看护手段**——确保项目过程不偏离目标、产出不偏离承诺、质量不偏离标准。

## 审计的治理定位

审计在治理体系中的位置：

```
Gate 检查（阶段转换关口）
    ↑
审计（交叉看护——在关键节点触发，发现偏差→形成任务→回到执行）
    ↑
日常执行（任务推进、证据记录、风险跟踪）
```

- **Gate**：阶段转换时的一次性质量门禁——"这个阶段可以结束了吗？"
- **审计**：持续进行的多维一致性检查——"我们还在正确的方向上吗？交付的东西用户能用吗？过程可信吗？"
- **审计不替代 Gate**：Gate 检查阶段产出物完整性，审计检查方向一致性、可用性、闭环质量

## 审计维度

审计维度按适用场景分为 3 类：**方向性维度**（检查是否向目标逼近）、**质量性维度**（检查执行过程是否可靠）、**体验性维度**（检查用户是否能用）。

### 方向性维度

#### D1 — 项目目标一致性审计（Goal Alignment Audit）

**审计问题**：当前完成的工作是否仍在向立项时定义的目标逼近？有没有偏离、范围漂移或优先级错位？

| 属性 | 内容 |
|------|------|
| 触发时机 | 每个 Gate 通过前（强制）、任何 P0 任务完成时（强制）、用户主动要求（按需） |
| 适用阶段 | 全部 11 阶段 |
| 检查项 | (1) 最近完成的 N 个任务是否可追溯到立项目标或 plan-tracker 中的明确需求；(2) plan-tracker 中是否有与立项目标无关但占用了 P0 资源的任务；(3) 决策记录中是否有改变原始目标方向但未在 plan-tracker 中反映的决策；(4) 范围是否发生了未经记录的漂移（对比立项时的"范围外"清单） |
| 产出 | 审计结论（ON-TRACK / DEVIATION-DETECTED / NEEDS-REALIGNMENT）+ 偏差清单 |
| 回写路径 | 偏差→decision-log；错位任务→plan-tracker 纠偏动作列 |

#### D2 — 用户使用角度审计（User Perspective Audit）

**审计问题**：从用户视角看，当前实现是否与 README 承诺一致？用户能否按照文档描述完成操作并获得预期结果？

| 属性 | 内容 |
|------|------|
| 触发时机 | 每个面向用户的功能完成时（强制）、每个里程碑结束时（强制）、外部验证前（强制） |
| 适用阶段 | 全部阶段，尤其是涉及 README/命令/安装流程的变更 |
| 检查项 | (1) README 的承诺是否能在当前实现中走通；(2) 用户按文档操作是否会遇到卡点（缺失文件、命令不可用、路径错误）；(3) agent 行为是否符合用户从 README 建立的预期；(4) 错误场景是否有明确的用户引导（不是静默失败） |
| 产出 | 审计结论 + 用户体验差距清单（按严重度：阻塞/困惑/失望） |
| 回写路径 | 差距→plan-tracker 新任务；误导性文档→AUDIT 任务 |

### 质量性维度

#### D3 — 阶段闭环质量审计（Stage Closure Quality Audit）

**审计问题**：阶段是否真正完成，还是只完成了文档层面的"表面闭环"？产出物是否能支撑下一阶段的工作？

| 属性 | 内容 |
|------|------|
| 触发时机 | 每个 Gate 通过前（强制——与 Gate 检查同步执行） |
| 适用阶段 | 全部 11 阶段 |
| 检查项 | (1) 阶段的产出物是否存在且内容充实（非空壳/非仅有标题）；(2) 退出条件 checklist 的每一项是否都有对应的证据（不是"已口头确认"）；(3) 阶段内的关键活动是否有遗漏（对照子工作流活动清单逐项检查）；(4) 本阶段的产出物是否已被下一阶段实际使用（验证产出物不是"写完就扔"） |
| 产出 | 阶段闭环评分（1-5）+ 未闭环项清单 |
| 回写路径 | 未闭环项→plan-tracker（阻塞下一阶段推进）；严重未闭环→risk-log |

#### D4 — 修改闭环标准审计（Change Closure Standards Audit）

**审计问题**：每一次修改是否真正闭环——有验证、有证据、有提交？还是"改了→声称完成→实际未验证"？

| 属性 | 内容 |
|------|------|
| 触发时机 | 每次 task 标记为"已完成"时（强制——与 Closure Follow-Through 规则并行）、每次 session 结束时（强制） |
| 适用阶段 | 全部阶段 |
| 检查项 | (1) 修改是否有对应的验证输出（build 通过/test 通过/命令输出——不是"已验证"三个字）；(2) 证据条目是否引用了持久化路径（非"会话上下文"）；(3) 修改涉及的关联文件是否都已被检查（修改 A 但 B 依赖 A，B 是否被验证？）；(4) commit message 是否准确反映了修改内容和原因 |
| 产出 | 修改闭环评分 + 未闭环修改清单 |
| 回写路径 | 未闭环修改→要求补充证据后才能标记为"已完成" |

### 体验性维度

#### D5 — 用户易用性审计（Usability Audit）

**审计问题**：一个不熟悉本项目的用户能否在合理时间内完成核心操作？摩擦点在哪里？

| 属性 | 内容 |
|------|------|
| 触发时机 | 外部验证时（强制）、用户反馈指出易用性问题时（按需）、每完成 3 个面向用户的功能后（建议） |
| 适用阶段 | 涉及安装/初始化/命令使用的阶段 |
| 检查项 | (1) 安装→初始化→首次使用的步骤数是否 ≤ README 承诺的"5 分钟"；(2) 用户在没有外部帮助的情况下能否完成初始化（测量：agent 需要澄清的问题数）；(3) 错误消息是否可操作（告诉用户"怎么做"而不只是"哪里错了"）；(4) 是否存在"只有开发者知道"的隐性知识依赖 |
| 产出 | 易用性评估报告 + 摩擦点清单（按严重度：阻塞/困难/困惑） |
| 回写路径 | 摩擦点→plan-tracker USABILITY 任务 |

#### D6 — 质量防护网看护预期审计（Quality Guardrail Audit）

**审计问题**：已建立的质量防护机制（CI/校验脚本/Gate 检查/check-governance）是否在真正起作用？是否有被绕过的路径？

| 属性 | 内容 |
|------|------|
| 触发时机 | 每个 Gate 通过前（强制）、CI 配置变更时（强制）、发现质量问题逃逸时（强制） |
| 适用阶段 | G4 之后（CI 建立后）的所有阶段 |
| 检查项 | (1) verify_workflow.py 是否每次修改后都通过；(2) check-governance 是否在 Gate 检查前被运行且结果为 PASSED；(3) 最近 N 次修改中是否有绕过防护网的案例（跳过 CI、跳过 Gate、跳过证据）；(4) 防护网自身的覆盖范围是否与项目规模匹配（轻量项目不需要全部防护，严格项目必须全覆盖）；(5) 防护网是否检测到了真实问题（如果从未检测到问题，说明防护网太松或未运行） |
| 产出 | 防护网健康度评估（EFFECTIVE / DEGRADED / BYPASSED）+ 防护缺口清单 |
| 回写路径 | 防护缺口→plan-tracker；防护网被绕过→risk-log |

### 维度汇总

| 维度 | 类别 | 触发频率 | 强制/建议 | 产出物 |
|------|------|---------|----------|--------|
| D1 目标一致性 | 方向 | 每 Gate + P0 完成 | **强制** | 偏差清单 |
| D2 用户视角 | 方向 | 里程碑 + 外部验证前 | **强制** | 体验差距清单 |
| D3 阶段闭环 | 质量 | 每 Gate 通过前 | **强制** | 闭环评分 + 未闭环项 |
| D4 修改闭环 | 质量 | 每 task + 每 session | **强制** | 未闭环修改清单 |
| D5 易用性 | 体验 | 外部验证 + 反馈驱动 | 强制（外部验证时）/建议 | 摩擦点清单 |
| D6 防护网看护 | 质量 | 每 Gate + CI 变更 | **强制** | 防护缺口清单 |

## 审计执行协议

### 执行顺序

当触发审计检查点时，按以下顺序执行：

1. **确认审计范围**：当前触发点适用哪些维度（查维度汇总表）
2. **逐维度执行**：对每个适用维度，按检查项列表逐项评估
3. **判定结果**：每个维度给出结论（PASS / ISSUES-FOUND / BLOCKED）
4. **记录发现**：发现的问题 MUST 写入对应回写路径（decision-log / plan-tracker / risk-log）
5. **输出审计报告**：按审计报告模板输出

### 审计报告模板

```
Audit Report — {trigger_description}

Dimensions audited: {dimension_list}

D1 Goal Alignment:     {PASSED / ISSUES / BLOCKED}
  {if issues}  - {finding_1}
  {if issues}  - {finding_2}
D2 User Perspective:   {PASSED / ISSUES / BLOCKED / SKIPPED}
  ...
D3 Stage Closure:      {PASSED / ISSUES / BLOCKED}
  ...
D4 Change Closure:     {PASSED / ISSUES / BLOCKED}
  ...
D5 Usability:          {PASSED / ISSUES / BLOCKED / SKIPPED}
  ...
D6 Quality Guardrail:  {PASSED / ISSUES / BLOCKED / SKIPPED}
  ...

Overall: {PASSED — no deviations / ISSUES FOUND — N issue(s) / BLOCKED — cannot proceed}

{if issues}
Findings disposition:
  → plan-tracker tasks: {task_ids}
  → decision-log: {decision_ids}
  → risk-log: {risk_ids}
```

### 审计发现→任务转化规则

| 发现严重度 | 转化规则 |
|-----------|---------|
| 阻塞级（Block） | 立即创建 P0 任务，当前 Gate 不得通过 |
| 偏差级（Deviation） | 创建 P0/P1 任务（按影响范围），Gate 可条件通过但必须记录 |
| 改进级（Improvement） | 创建 P1/P2 任务，不阻塞 Gate |
| 注意级（Observation） | 记录到 decision-log，不强制创建任务 |

### 审计自身质量保证（Meta-Audit）

审计机制本身也接受审计（D4 修改闭环维度 + D6 防护网维度）：

- 每次审计完成后，审计报告 MUST 写入 evidence-log
- 审计发现的任务转化率 MUST ≥ 80%（发现的偏差级及以上问题必须全部转化为任务）
- 如果审计连续 3 次触发但均未发现问题 → 审计维度可能需要调整（可能检查项太松或审计未真正执行）

## Profile 对审计的影响

| Profile | 审计强度 | 说明 |
|---------|---------|------|
| lightweight | 精简审计 | 仅 D1（目标一致性）和 D4（修改闭环）在 Gate 时强制；其他维度按需 |
| standard | 标准审计 | D1/D3/D4/D6 在 Gate 时强制；D2 在里程碑时强制；D5 在外部验证时强制 |
| strict | 增强审计 | 全部 6 个维度在 Gate 时强制；D2/D5 在每个 P0 任务完成时也需执行 |

## Agent 审计纪律

执行审计时，agent MUST：

1. **以用户视角审视**：不是"代码写对了没"，是"用户拿这个东西能用吗、方向对吗"
2. **基于事实而非印象**：每个审计判定必须引用具体文件、具体行数、具体命令输出
3. **审计结论必须可追溯**：审计报告中的每一条发现必须有对应的证据位置
4. **区分"建议"和"缺陷"**：不要把所有发现都标记为阻塞（Block），精确评估严重度
5. **审计完成后必须回写**：不能"审完就结束"——发现必须转化为 plan-tracker/decision-log/risk-log 中的条目
