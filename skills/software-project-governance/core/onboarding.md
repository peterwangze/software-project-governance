# 中途接入协议

本文件定义项目已在进行中时如何接入 `software-project-governance` 工作流。

## 目标

让已在进行中的项目（可能已处于开发、测试或任何阶段）能够平滑接入工作流，不需要从立项阶段重新开始。

## 接入流程

### Step 1：声明当前状态

用户需要提供以下信息：

1. **项目名称和目标**：一句话描述项目在做什么
2. **当前所处阶段**：对照 `lifecycle.md` 的 11 个阶段，声明项目当前处于哪个阶段
3. **已完成的关键产出物**：列出已完成的主要产出物（如架构文档、代码仓库、测试用例等）
4. **已知风险**：当前已知的主要风险或阻塞
5. **选择 profile**：对照 `profiles.md` 选择 lightweight / standard / strict

### Step 2：补齐最小记录

根据项目 profile 和当前阶段，补齐以下最小记录：

#### 所有 profile 都必须补齐

- 项目总览面板（项目名称、当前阶段、总任务数）
- 至少 3 条当前阶段的任务项
- 当前阶段的 Gate 检查结果（显式声明通过/未通过/有条件通过）

#### standard 和 strict profile 额外要求

- 当前阶段的关键决策记录（至少 1 条）
- 当前阶段的风险记录（至少 1 条）
- 当前阶段的证据记录（至少 1 条）

#### strict profile 额外要求

- 前置所有阶段的最小证据回顾（不是从零补齐，而是承认"已完成并记录关键事实"）
- 每个前置阶段至少 1 条决策和 1 条风险

### Step 3：前置阶段处理

对于当前阶段之前的所有阶段，不要求从零执行，但要求：

1. **承认已完成**：前置阶段的 Gate 标记为"接入时自动通过"
2. **记录关键事实**：每个前置阶段至少记录 1 条关键决策（如"为什么选了这个技术栈"）
3. **不补齐全部细节**：前置阶段的记录只要求覆盖关键决策点，不要求完整

处理方式：

```
前置阶段 Gate 状态：
  - 接入前已完成的阶段：标记为 "passed-on-entry"
  - 接入时处于的阶段：标记为 "in-progress"
  - 接入后未到的阶段：标记为 "pending"
```

### Step 4：进入当前阶段子工作流

完成最小记录补齐后，立即进入当前阶段的子工作流：

1. 读取 `stages/<current-stage>/sub-workflow.md`
2. 检查当前阶段的进入条件是否满足
3. 如果进入条件未完全满足，补齐缺失项或记录为风险
4. 开始执行当前阶段的活动清单

## 最小接入路径

### lightweight profile 的最小路径

1. 声明项目名称、目标、当前阶段
2. 在计划跟踪表写入当前阶段的 3 条任务
3. 执行当前阶段 Gate 检查
4. 开始工作

### standard profile 的最小路径

1. 完成上述 lightweight 路径
2. 补齐当前阶段至少 1 条决策、1 条风险、1 条证据
3. 对前置阶段各补 1 条关键决策
4. 开始工作

### strict profile 的最小路径

1. 完成上述 standard 路径
2. 对每个前置阶段各补 1 条决策 + 1 条风险
3. 对当前阶段补齐至少 2 条证据
4. 正式进入当前阶段子工作流

## 特殊场景

### 项目处于维护阶段

如果项目已经处于维护/演进阶段，可以只启用维护阶段子工作流，不要求补齐全流程记录。使用 on-demand 触发模式。

### 项目处于多个阶段之间

如果项目的工作同时涉及多个阶段（如同时在开发和测试），声明主要阶段为当前阶段，次要阶段标记为"并行活跃"。

### 不确定当前阶段

如果用户无法判断当前处于哪个阶段，工作流提供一个阶段判定 checklist：
- 逐个读取阶段进入条件和退出条件
- 找到"上一个退出条件已满足、下一个进入条件未满足"的阶段
- 将该阶段判定为当前阶段

## 与其他规则的关系

- `lifecycle.md`：阶段列表以 lifecycle 为准
- `profiles.md`：profile 决定接入时的最小记录要求
- `stage-gates.md`：接入时前置 Gate 的处理方式以 stage-gates 为准
